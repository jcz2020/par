# MiniMax-M3 结构化输出能力评估报告（修正版）

> **测试日期**: 2026-06-20
> **模型**: MiniMax-M3
> **API端点**: `https://api.minimaxi.com/v1`（OpenAI兼容）
> **测试工具**: PAR SDK v0.4.8-beta `Engine.run_structured`（降级路径）
> **测试规模**: 6种场景 × 20轮 = **120次真实API调用**
> **修正说明**: 初版报告因未剥离 `<think>` 标签导致M3成功率被误判为40.8%。修正后（剥离think + max_tokens=2000），实际成功率为**100%**。

---

## 一、测试目的

评估 MiniMax-M3 在结构化输出场景下的能力表现。

---

## 二、测试设计

### 2.1 测试路径

使用 PAR SDK 的降级路径（prompt指令 + JSON提取 + Schema校验 + 修复循环），不使用供应商原生结构化输出API。

### 2.2 LLM收到的输入

**System Prompt（第1次尝试）**:
```
You are a helpful assistant.

You MUST respond with a valid JSON object matching this exact schema: <schema_json>
Do not include any text outside the JSON object.
```

**User Message**: 各场景的具体提示词（见下文）

**修复反馈（第2/3次尝试）**:
- JSON解析失败: `Your previous response was not valid JSON: <错误>. Please respond with valid JSON matching the schema.`
- Schema校验失败: `Schema violation: <错误>. Please respond with JSON matching the schema.`

**参数**: temperature=0.0, max_tokens=2000, max_repair_attempts=3

### 2.3 六种测试场景

#### 场景1: country-fr — 基础格式遵循

**提示词**: `Tell me about France in JSON.`

**Schema**: `{"type":"object","properties":{"name":{"type":"string"},"capital":{"type":"string"}},"required":["name","capital"]}`

**预期输出**: `{"name":"France","capital":"Paris"}`

**考核点**: 模型能否按指令返回一个包含2个string字段的简单JSON对象

#### 场景2: country-jp — 一致性验证

**提示词**: `Tell me about Japan in JSON.`

**Schema**: 与场景1相同

**预期输出**: `{"name":"Japan","capital":"Tokyo"}`

**考核点**: 同一schema、不同主题，成功率是否一致

#### 场景3: movie — 混合数据类型

**提示词**: `Recommend a sci-fi movie in JSON.`

**Schema**: `{"type":"object","properties":{"title":{"type":"string"},"year":{"type":"integer"},"rating":{"type":"number"}},"required":["title","year"]}`

**预期输出**: `{"title":"Blade Runner 2049","year":2017,"rating":8.0}`

**考核点**: 能否在同一对象中正确使用 string、integer、number 三种类型

#### 场景4: book — 混合类型一致性

**提示词**: `Recommend a programming book in JSON.`

**Schema**: 与场景3相同

**预期输出**: `{"title":"Clean Code","year":2008,"rating":4.7}`

**考核点**: 同一schema、不同领域，类型准确性是否一致

#### 场景5: todo — 复合结构（数组包装）

**提示词**: `Give me a 3-item todo list in JSON.`

**Schema**: `{"type":"object","properties":{"items":{"type":"array"},"count":{"type":"integer"}},"required":["items","count"]}`

**预期输出**: `{"items":["Buy groceries","Finish report","Call mom"],"count":3}`

**考核点**: 能否返回一个包含数组字段的对象（而非直接返回裸数组）

#### 场景6: word — 枚举约束

**提示词**: `Classify the word 'running' in JSON.`

**Schema**: `{"type":"object","properties":{"word":{"type":"string"},"category":{"type":"string","enum":["noun","verb","adjective"]}},"required":["word","category"]}`

**预期输出**: `{"word":"running","category":"verb"}`

**考核点**: 能否同时遵循"返回JSON格式"和"category值只能是noun/verb/adjective之一"两个约束

### 2.4 考核指标

| 指标 | 定义 |
|---|---|
| 成功率 | 20轮中返回有效JSON且通过Schema校验的比例 |
| Schema校验通过率 | 成功调用中Schema校验通过的比例（100% = 零误报） |
| 平均尝试次数 | 成功调用中平均需要几次LLM调用 |

## 四、修正过程与根因分析

### 4.1 问题发现

初版测试中，MiniMax-M3成功率仅40.8%，远低于GLM-5.2的100%。初版报告据此判定M3"不适合结构化输出"。

用户提出假设：**M3效果差是否因为 `<think>` 标签没有被去掉？**

### 4.2 假设验证

通过curl直接调用API，检查M3的原始响应：

```
<think>The user wants information about France in JSON format.
I need to respond with a valid JSON object matching the schema.</think>

{"name":"France","capital":"Paris"}
```

**JSON确实存在**，就在 `<think>` 块之后。但PAR的 `Json_extract` 未能正确提取，原因是think块内的花括号字符干扰了平衡括号扫描器。

### 4.3 修正方法

在 `complete_fn` 返回响应前，自动剥离 `<think>...</think>` 块：

```ocaml
let strip_think text =
  match find_sub text "</think>" with
  | Some i -> String.trim (String.sub text (i + 8) (String.length text - i - 8))
  | None -> text
```

同时将 `max_tokens` 从500提升至2000。

### 4.4 修正效果

修正后，M3在所有6种场景、20轮120次调用中**全部首次尝试即成功**（avg attempts = 1.0）。之前的"M3能力差"结论完全错误——这是PAR工具链未正确处理 `<think>` 标签导致的误判。

### 4.5 修正前失败模式（已消除）

| 失败类型 | 次数 | 占比 | 根因 |
|---|---|---|---|
| `no valid JSON found` | 44次 | 62% | think块内花括号干扰括号扫描器 |
| `missing required fields` | 14次 | 20% | JSON被max_tokens截断 |
| `expected object, got array` | 13次 | 18% | think块内讨论了数组结构，扫描器提取错误片段 |

## 五、结论

### MiniMax-M3 结构化输出能力评级（修正后）

| 维度 | 评级 | 依据 |
|---|---|---|
| **指令遵循** | 5/5 优秀 | 100%首次成功返回有效JSON |
| **Schema复杂度适应** | 5/5 极强 | 简单对象/数组/Enum全部100% |
| **数据类型准确性** | 5/5 优秀 | 100%类型正确，零误报 |
| **稳定性** | 5/5 优秀 | 20轮零波动 |

### 修正前结论（已撤回）

初版报告判定M3"不适合结构化输出，成功率仅40.8%"。**此结论是错误的。** 40.8%的成功率是PAR工具链bug（未剥离 `<think>` 标签）造成的假象。修正后M3的实际结构化输出能力为100%，与GLM-5.2和MiniMax-M2.7持平。

### 对PAR SDK的建议

`<think>` 标签剥离逻辑必须加入PAR的 `Json_extract` 生产代码。这不仅影响MiniMax系列，还影响所有使用推理链的模型（DeepSeek-R1、未来的thinking模型等）。
