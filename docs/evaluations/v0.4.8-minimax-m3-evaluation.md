# MiniMax-M3 结构化输出能力评估报告

> **测试日期**: 2026-06-20
> **模型**: MiniMax-M3
> **API端点**: `https://api.minimaxi.com/v1`（OpenAI兼容）
> **测试工具**: PAR SDK v0.4.8-beta `Engine.run_structured`（降级路径）
> **测试规模**: 6种场景 × 20轮 = **120次真实API调用**

---

## 一、测试目的

评估 MiniMax-M3 在结构化输出场景下的能力表现，具体回答以下问题：

1. MiniMax-M3 能否按 JSON Schema 指令返回结构化数据？
2. 不同复杂度的 Schema 对成功率的影响有多大？
3. 修复循环（最多3次重试）对 MiniMax-M3 的纠偏效果如何？
4. MiniMax-M3 的主要失败模式是什么？

---

## 二、测试设计

### 2.1 测试路径

使用 PAR SDK 的降级路径（prompt指令 + JSON提取 + Schema校验 + 修复循环），不使用供应商原生结构化输出API。MiniMax-M3 的 OpenAI 兼容端点静默忽略 `response_format` 字段，因此所有调用走降级路径。

### 2.2 LLM收到的输入

**System Prompt（第1次尝试）**:
```
You are a helpful assistant.

You MUST respond with a valid JSON object matching this exact schema: <schema_json>
Do not include any text outside the JSON object.
```

**User Message**: 各场景的具体提示词（见下文）

**修复反馈（第2/3次尝试）**:
- JSON解析失败: `Your previous response was not valid JSON: <错误>. Please respond with valid JSON matching the schema.`
- Schema校验失败: `Schema violation: <错误>. Please respond with JSON matching the schema.`

**参数**: temperature=0.0, max_tokens=500, max_repair_attempts=3

### 2.3 六种测试场景

#### 场景1: country-fr — 基础格式遵循

**提示词**: `Tell me about France in JSON.`

**Schema**:
```json
{"type":"object","properties":{"name":{"type":"string"},"capital":{"type":"string"}},"required":["name","capital"]}
```

**预期输出**: `{"name":"France","capital":"Paris"}`

**考核点**: 模型能否按指令返回一个包含2个string字段的简单JSON对象

---

#### 场景2: country-jp — 一致性验证

**提示词**: `Tell me about Japan in JSON.`

**Schema**: 与场景1相同

**预期输出**: `{"name":"Japan","capital":"Tokyo"}`

**考核点**: 同一schema、不同主题，成功率是否一致

---

#### 场景3: movie — 混合数据类型

**提示词**: `Recommend a sci-fi movie in JSON.`

**Schema**:
```json
{"type":"object","properties":{"title":{"type":"string"},"year":{"type":"integer"},"rating":{"type":"number"}},"required":["title","year"]}
```

**预期输出**: `{"title":"Blade Runner 2049","year":2017,"rating":8.0}`

**考核点**: 能否在同一对象中正确使用 string、integer、number 三种类型（year必须是整数2017而非字符串"2017"）

---

#### 场景4: book — 混合类型一致性

**提示词**: `Recommend a programming book in JSON.`

**Schema**: 与场景3相同

**预期输出**: `{"title":"Clean Code","year":2008,"rating":4.7}`

**考核点**: 同一schema、不同领域，类型准确性是否一致

---

#### 场景5: todo — 复合结构（数组包装）

**提示词**: `Give me a 3-item todo list in JSON.`

**Schema**:
```json
{"type":"object","properties":{"items":{"type":"array"},"count":{"type":"integer"}},"required":["items","count"]}
```

**预期输出**: `{"items":["Buy groceries","Finish report","Call mom"],"count":3}`

**考核点**: 能否返回一个**包含数组字段的对象**（而非直接返回裸数组）

---

#### 场景6: word — 枚举约束

**提示词**: `Classify the word 'running' in JSON.`

**Schema**:
```json
{"type":"object","properties":{"word":{"type":"string"},"category":{"type":"string","enum":["noun","verb","adjective"]}},"required":["word","category"]}
```

**预期输出**: `{"word":"running","category":"verb"}`

**考核点**: 能否同时遵循"返回JSON格式"和"category值只能是noun/verb/adjective之一"两个约束

---

### 2.4 考核指标

| 指标 | 定义 |
|---|---|
| 成功率 | 20轮中返回有效JSON且通过Schema校验的比例 |
| Schema校验通过率 | 成功调用中Schema校验通过的比例（100% = 零误报） |
| 平均尝试次数 | 成功调用中平均需要几次LLM调用 |
| 首次成功率 | 第一次调用就成功的比例（估算） |

## 四、失败模式分析

### 4.1 失败类型分布（71次失败）

| 失败类型 | 次数 | 占比 | 典型错误信息 |
|---|---|---|---|
| `no valid JSON found` | 44次 | **62%** | 3次尝试都只返回 `<think>` 标签+自由文本，无任何JSON |
| `missing required fields` | 14次 | **20%** | 返回了JSON但缺少必填字段（如只返回了 `word` 没有 `category`） |
| `expected object, got array` | 13次 | **18%** | 返回了裸数组 `["a","b","c"]` 而非 `{"items":["a","b","c"],"count":3}` |

### 4.2 间歇性崩溃现象

Round 6（1/6成功）、Round 14（**0/6成功**）、Round 18（1/6成功）出现全盘崩溃。这不是某个场景的问题，而是模型在特定时间点完全"拒绝"输出JSON格式。这表明 MiniMax-M3 存在**不稳定的指令遵循**——有时能理解JSON格式要求，有时完全无视。

### 4.3 Schema回显异常（Round 13）

Round 13 的 country-fr 返回了 `{"name":"string","capital":"string"}` — 模型把 Schema 定义中的类型名 `"string"` 当成了数据值。这通过了 Schema 校验（因为 `"string"` 确实是 string 类型），但数据内容完全错误。这种"格式正确但语义错误"的模式是危险的——用户可能拿到看似有效但实际无意义的数据。

## 五、修复循环效果评估

| 指标 | 无修复循环（估算） | 有修复循环（实测） | 提升 |
|---|---|---|---|
| 成功率 | ~20% | **40.8%** | +20.8个百分点 |
| 平均尝试次数 | 1.0 | 2.2 | 每次成功需多1.2次调用 |

修复循环对 MiniMax-M3 有效果但**效果有限**：
- 44/71次失败是"3次尝试都无JSON"——修复反馈无法让模型改变输出格式
- 14/71次失败是"字段缺失"——修复后仍不完整
- 只有约20%的失败被修复循环成功纠正

## 六、结论

### MiniMax-M3 结构化输出能力评级

| 维度 | 评级 | 依据 |
|---|---|---|
| **指令遵循** | ⭐☆☆☆☆ 差 | 首次调用仅~20%返回有效JSON |
| **Schema复杂度适应** | ⭐☆☆☆☆ 很弱 | 简单对象50% → 数组20% → Enum40% |
| **数据类型准确性** | ⭐⭐⭐⭐☆ 良好 | 成功时100%类型正确 |
| **修复循环配合度** | ⭐⭐☆☆☆ 弱 | 修复仅提升20pp |
| **稳定性** | ⭐☆☆☆☆ 差 | Round 6/14/18出现间歇性全盘崩溃 |

### 总结

MiniMax-M3 **不适合用于结构化输出场景**。40.8%的成功率意味着每5次调用中有3次失败。即使有修复循环（3次重试），仍有59.2%的调用无法得到有效结果。主要瓶颈不在"不会用JSON"（类型准确性100%），而在"不愿意返回JSON格式"（62%的失败是完全没有JSON输出）。

### 建议

1. **需要结构化输出时避免使用 MiniMax-M3**
2. 如果必须使用，设 `max_repair_attempts=5` 以上，并接受约40%的失败率
3. 优先考虑 MiniMax-M2.7（82.5%成功率）或 GLM-5.2（100%成功率）
