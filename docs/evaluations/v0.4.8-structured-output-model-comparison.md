# PAR v0.4.8 结构化输出跨模型能力评估报告

> **测试日期**: 2026-06-20
> **测试工具**: PAR SDK v0.4.8-beta `Engine.run_structured`（降级路径）
> **测试规模**: 3个模型 × 6种场景 × 20轮 = **360次真实API调用**

---

## 一、测试设计

### 1.1 测试路径

所有测试走 PAR 的**降级路径**（prompt指令 + JSON提取 + Schema校验 + 修复循环），不使用供应商的原生结构化输出API。原因：

- MiniMax 的两个模型不支持 OpenAI `response_format` 字段（静默忽略）
- GLM-5.2 的智谱端点同样使用降级路径
- 统一走降级路径可以**公平对比模型本身的能力**，而非供应商API的实现差异

### 1.2 LLM收到的完整输入

每次调用，LLM收到以下对话：

**System Prompt（第1次尝试）**:
```
You are a helpful assistant.

You MUST respond with a valid JSON object matching this exact schema: <schema_json>
Do not include any text outside the JSON object.
```

**User Message**:
```
<具体场景的提示词>
```

**修复反馈（第2次及之后的尝试，仅在上一次失败时追加）**:

- JSON解析失败时: `Your previous response was not valid JSON: <错误详情>. Please respond with valid JSON matching the schema.`
- Schema校验失败时: `Schema violation: <错误详情>. Please respond with JSON matching the schema.`

**参数**:
- `temperature = 0.0`（确定性输出，减少随机性）
- `max_tokens = 500`
- `max_repair_attempts = 3`（最多3次尝试）

### 1.3 6种测试场景详解

---

#### 场景1: country-fr（国家信息 — 法国）

**考核能力**: 基础格式遵循 — 模型能否按指令返回一个简单的JSON对象

**User Message**:
```
Tell me about France in JSON.
```

**JSON Schema**:
```json
{
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "capital": {"type": "string"}
  },
  "required": ["name", "capital"]
}
```

**预期输出**:
```json
{"name": "France", "capital": "Paris"}
```

**考核点**:
1. 模型是否返回JSON格式（而非自由文本）
2. JSON中是否包含 `name` 和 `capital` 两个必填字段
3. 两个字段的值是否为 string 类型

---

#### 场景2: country-jp（国家信息 — 日本）

**考核能力**: 一致性 — 同一schema、不同主题，成功率是否稳定

**User Message**:
```
Tell me about Japan in JSON.
```

**JSON Schema**: 与场景1相同

**预期输出**:
```json
{"name": "Japan", "capital": "Tokyo"}
```

**考核点**: 与场景1相同，但主题不同。如果场景1成功率高但场景2低，说明模型对特定主题有偏见。

---

#### 场景3: movie（电影推荐）

**考核能力**: 混合数据类型 — 能否在同一对象中正确使用 string、integer、number 三种类型

**User Message**:
```
Recommend a sci-fi movie in JSON.
```

**JSON Schema**:
```json
{
  "type": "object",
  "properties": {
    "title": {"type": "string"},
    "year": {"type": "integer"},
    "rating": {"type": "number"}
  },
  "required": ["title", "year"]
}
```

**预期输出**:
```json
{"title": "Blade Runner 2049", "year": 2017, "rating": 8.0}
```

**考核点**:
1. `title` 是 string 类型
2. `year` 是 integer 类型（不是 string `"2017"`）
3. `rating` 是 number 类型（可以是浮点数如 8.7）
4. 必填字段 `title` 和 `year` 都存在（`rating` 可选）

**难点**: 模型容易把 `year` 返回为字符串 `"2017"` 而非整数 `2017`。这是JSON类型系统的常见错误。

---

#### 场景4: book（书籍推荐）

**考核能力**: 混合类型一致性 — 与场景3相同的schema，不同领域

**User Message**:
```
Recommend a programming book in JSON.
```

**JSON Schema**: 与场景3相同

**预期输出**:
```json
{"title": "Clean Code: A Handbook of Agile Software Craftsmanship", "year": 2008, "rating": 4.7}
```

**考核点**: 与场景3相同。如果场景3成功但场景4失败，说明模型对特定领域理解不足。

---

#### 场景5: todo（待办列表）

**考核能力**: 复合结构 — 能否返回一个**包含数组字段的对象**

**User Message**:
```
Give me a 3-item todo list in JSON.
```

**JSON Schema**:
```json
{
  "type": "object",
  "properties": {
    "items": {"type": "array"},
    "count": {"type": "integer"}
  },
  "required": ["items", "count"]
}
```

**预期输出**:
```json
{"items": ["Buy groceries", "Finish project report", "Call mom"], "count": 3}
```

**考核点**:
1. 返回的是一个**对象**（不是裸数组）
2. `items` 字段的值是一个**数组**
3. `count` 字段的值是**整数**
4. 两个必填字段都存在

**难点**: 这是三个模型差异最大的场景。MiniMax系列倾向于直接返回裸数组 `["task1", "task2", "task3"]`，而非包装在对象里。GLM-5.2则每次都能正确返回 `{"items": [...], "count": 3}` 的包装结构。

---

#### 场景6: word（词语分类）

**考核能力**: 约束遵循 — 能否遵守枚举值限制

**User Message**:
```
Classify the word 'running' in JSON.
```

**JSON Schema**:
```json
{
  "type": "object",
  "properties": {
    "word": {"type": "string"},
    "category": {
      "type": "string",
      "enum": ["noun", "verb", "adjective"]
    }
  },
  "required": ["word", "category"]
}
```

**预期输出**:
```json
{"word": "running", "category": "verb"}
```

**考核点**:
1. `word` 字段值是字符串
2. `category` 字段值必须是 `"noun"`、`"verb"` 或 `"adjective"` 之一（不能是其他值如 `"adverb"` 或 `"participle"`）
3. 两个必填字段都存在

**难点**: 模型需要同时理解"返回JSON格式"和"category值只能是三个选项之一"两个约束。这个双重约束对指令遵循能力弱 的模型是重大挑战。

---

### 1.4 考核指标定义

| 指标 | 定义 | 意义 |
|---|---|---|
| **成功率** | 20轮中返回有效JSON且通过Schema校验的比例 | 模型结构化输出的整体可靠性 |
| **Schema校验通过率** | 成功的调用中Schema校验通过的比例 | 数据正确性（零误报 = 100%意味着没有"看起来对但实际不对"的结果） |
| **平均尝试次数** | 成功调用中平均需要几次LLM调用 | 修复循环的必要性和效率 |
| **首次成功率** | 第一次调用就成功的比例（估算自平均尝试次数） | 不依赖修复循环时的"原始"能力 |

### 1.5 测试过程

#### 执行方式

测试通过一个自动化 OCaml 脚本（`examples/verify_structured.ml`）执行。脚本对每个模型跑 20 轮，每轮依次执行 6 个场景，共计 120 次结构化输出调用。三个模型使用完全相同的脚本，仅切换环境变量（API Key、Base URL、模型名）。

```bash
# GLM-5.2
OPENAI_API_KEY="..." OPENAI_BASE_URL="https://open.bigmodel.cn/api/coding/paas/v4" \
  dune exec examples/verify_structured.exe

# MiniMax-M2.7 / M3
OPENAI_API_KEY="..." OPENAI_BASE_URL="https://api.minimaxi.com/v1" \
  dune exec examples/verify_structured.exe
```

#### 单次调用的完整流程

每次场景测试，脚本内部执行以下步骤：

```
1. 构建对话
   System: "You are a helpful assistant.
            You MUST respond with a valid JSON object matching this exact schema: <schema>
            Do not include any text outside the JSON object."
   User:   "<场景提示词，如 Tell me about France in JSON.>"

2. 调用 LLM（第1次尝试）
   → 通过 OpenAI 兼容 API 发送 HTTP POST 请求
   → 等待模型返回

3. 处理响应
   3a. 从响应文本中提取 JSON（Json_extract.extract_json_from_text）
       → 尝试直接解析、剥离 markdown fence、扫描平衡括号
   3b. 如果提取失败：
       → 追加修复反馈："Your previous response was not valid JSON: <错误>. 
         Please respond with valid JSON matching the schema."
       → 回到步骤2（第2次尝试）
   3c. 如果提取成功：
       → 用 Schema 校验 JSON（Validation.validate_tool_input_result）
   3d. 如果校验失败：
       → 追加修复反馈："Schema violation: <错误>. 
         Please respond with JSON matching the schema."
       → 回到步骤2（第2次尝试）
   3e. 如果校验成功：
       → 记录成功（attempt = 当前尝试次数, value = 提取的JSON）

4. 如果3次尝试都失败：
   → 记录失败（error = 最后一次错误信息）

5. 输出一行结果：
   country-fr  OK  attempts=2  valid=true  value={"name":"France","capital":"Paris"}
```

#### 耗时

| 模型 | 120次调用总耗时 | 平均单次调用 | 说明 |
|---|---|---|---|
| GLM-5.2 | ~8分钟 | ~4秒 | 几乎不需要修复循环，每次1次调用就完成 |
| MiniMax-M2.7 | ~25分钟 | ~12秒 | 平均2.2次尝试，每次约5-6秒 |
| MiniMax-M3 | ~25分钟 | ~12秒 | 平均2.2次尝试（含失败的3次耗尽），每次约5-6秒 |

GLM-5.2 因为首次成功率极高（~99%），绝大多数场景只需1次 LLM 调用，总调用量约120次。MiniMax 两个版本因为修复循环频繁触发，实际 LLM 调用量约260-280次（120次成功+失败的重试），所以耗时约为 GLM-5.2 的3倍。

#### 公平性保障

| 控制项 | 设置 | 说明 |
|---|---|---|
| 测试脚本 | 完全相同（同一份 `verify_structured.ml`） | 仅改环境变量，不改代码 |
| Schema | 6种完全相同 | 三个模型面对同样的结构化要求 |
| 提示词 | 完全相同 | 同一个 system prompt + user message |
| 修复循环 | `max_repair_attempts = 3` | 三个模型都给最多3次机会 |
| Temperature | `0.0` | 确定性输出，减少随机波动 |
| Max tokens | `500` | 足够返回完整JSON |
| 调用路径 | 全部走降级路径 | 不使用任何供应商的原生结构化API，排除API实现差异 |

#### 数据采集

每轮每个场景输出一行结构化结果，包含4个字段：
- **状态**: OK / FAIL
- **尝试次数**: 1-3（仅OK时有意义）
- **Schema校验**: true / false（仅OK时有意义）
- **返回值**: 提取到的JSON原文（仅OK时）

脚本最后自动汇总：总成功率、Schema通过率、平均尝试次数、按场景分组的统计。

### 2.1 修正前 vs 修正后

测试过程中发现关键问题：MiniMax系列模型在输出JSON前会先输出 `<think>...</think>` 推理块，PAR的 `Json_extract` 未正确处理此格式，导致大量误判失败。修正方法：在JSON提取前自动剥离 `<think>` 标签 + 将 `max_tokens` 从500提升至2000。

## 三、关键发现：think标签是唯一瓶颈

### 3.1 问题定位

首轮测试（不剥离think标签）中，MiniMax-M3成功率仅40.8%，MiniMax-M2.7仅82.5%。当时分析认为是"模型指令遵循能力差"。

**这个判断是完全错误的。**

通过检查原始API响应发现，MiniMax模型每次都正确返回了JSON——只是JSON前面包裹了 `<think>推理过程</think>` 块。PAR的 `Json_extract` 在处理这种格式时失败，原因是：

1. **think块内的花括号干扰**：模型在推理过程中可能引用schema结构（如"我需要返回 `{"name": ..., "capital": ...}`"），这些花括号误导了平衡括号扫描器
2. **max_tokens不足**：think块占用token预算，复杂场景下JSON可能被截断（`finish_reason: length`）

### 3.2 修正方法

在 `complete_fn` 返回响应前，自动剥离 `<think>...</think>` 块：

```ocaml
let strip_think text =
  match find_sub text "</think>" with
  | Some i -> String.trim (String.sub text (i + 8) (String.length text - i - 8))
  | None -> text
```

同时将 `max_tokens` 从500提升至2000，确保think + JSON都有足够空间。

### 3.3 修正效果

修正后，三个模型在所有6种场景、20轮120次调用中**全部达到100%成功率**。之前观察到的"M3不如M2.7"、"数组场景困难"、"Enum约束遵循差"等现象**全部消失**——这些都是工具链缺陷造成的假象，而非模型能力差异。

## 六、结论

### 对PAR SDK的结论

1. **`<think>`标签剥离是必须的生产代码修复**。PAR的 `Json_extract` 必须在提取JSON前自动剥离 `<think>...</think>` 块，否则对所有使用推理链的模型（MiniMax、DeepSeek-R1等）都会严重失效。

2. **修复循环仍有价值**：即使在剥离think标签后，M2.7的word场景仍有少数情况需要2次尝试（avg_att=1.5），修复循环作为安全网仍然必要。

3. **max_tokens建议值**：结构化输出场景建议 `max_tokens >= 2000`，为推理链 + JSON输出预留足够空间。

### 对模型选型的结论

**三个模型在结构化输出场景下能力相当，均可达到100%成功率。** 之前的性能差异完全是PAR工具链未正确处理 `<think>` 标签所致。

选型建议：
- **GLM-5.2**：不使用think标签，输出最干净，首次成功率最高
- **MiniMax-M2.7/M3**：使用think标签，但只要PAR正确剥离，表现与GLM-5.2一致
- 不存在"M3不如M2.7"的问题——这是工具链bug造成的假象

### 对测试方法论的教训

本轮测试的最大价值不是模型对比结论，而是**暴露了一个关键的工具链缺陷**。如果没有用户提出的"是不是think标签的问题"这个假设，我们可能会带着"M3能力差"的错误结论发布报告。

**教训：在归因模型性能差异之前，必须排除工具链干扰。**
