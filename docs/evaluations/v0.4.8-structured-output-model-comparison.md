# PAR v0.4.8 结构化输出跨模型能力评估报告

> **测试日期**: 2026-06-20
> **测试工具**: PAR SDK v0.4.8-beta `Engine.run_structured`（降级路径）
> **测试规模**: 3个模型 × 6种场景 × 20轮 = **360次真实API调用**

---

## 一、测试设计

### 1.1 测试路径

所有测试走 PAR 的**降级路径**（prompt指令 + JSON提取 + Schema校验 + 修复循环），不使用供应商的原生结构化输出API。原因：

- MiniMax 的两个模型不支持 OpenAI `response_format` 字段（静默忽略）
- GLM-5.2 的智谱端点同样使用降级路径
- 统一走降级路径可以**公平对比模型本身的能力**，而非供应商API的实现差异

### 1.2 LLM收到的完整输入

每次调用，LLM收到以下对话：

**System Prompt（第1次尝试）**:
```
You are a helpful assistant.

You MUST respond with a valid JSON object matching this exact schema: <schema_json>
Do not include any text outside the JSON object.
```

**User Message**:
```
<具体场景的提示词>
```

**修复反馈（第2次及之后的尝试，仅在上一次失败时追加）**:

- JSON解析失败时: `Your previous response was not valid JSON: <错误详情>. Please respond with valid JSON matching the schema.`
- Schema校验失败时: `Schema violation: <错误详情>. Please respond with JSON matching the schema.`

**参数**:
- `temperature = 0.0`（确定性输出，减少随机性）
- `max_tokens = 500`
- `max_repair_attempts = 3`（最多3次尝试）

### 1.3 6种测试场景详解

---

#### 场景1: country-fr（国家信息 — 法国）

**考核能力**: 基础格式遵循 — 模型能否按指令返回一个简单的JSON对象

**User Message**:
```
Tell me about France in JSON.
```

**JSON Schema**:
```json
{
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "capital": {"type": "string"}
  },
  "required": ["name", "capital"]
}
```

**预期输出**:
```json
{"name": "France", "capital": "Paris"}
```

**考核点**:
1. 模型是否返回JSON格式（而非自由文本）
2. JSON中是否包含 `name` 和 `capital` 两个必填字段
3. 两个字段的值是否为 string 类型

---

#### 场景2: country-jp（国家信息 — 日本）

**考核能力**: 一致性 — 同一schema、不同主题，成功率是否稳定

**User Message**:
```
Tell me about Japan in JSON.
```

**JSON Schema**: 与场景1相同

**预期输出**:
```json
{"name": "Japan", "capital": "Tokyo"}
```

**考核点**: 与场景1相同，但主题不同。如果场景1成功率高但场景2低，说明模型对特定主题有偏见。

---

#### 场景3: movie（电影推荐）

**考核能力**: 混合数据类型 — 能否在同一对象中正确使用 string、integer、number 三种类型

**User Message**:
```
Recommend a sci-fi movie in JSON.
```

**JSON Schema**:
```json
{
  "type": "object",
  "properties": {
    "title": {"type": "string"},
    "year": {"type": "integer"},
    "rating": {"type": "number"}
  },
  "required": ["title", "year"]
}
```

**预期输出**:
```json
{"title": "Blade Runner 2049", "year": 2017, "rating": 8.0}
```

**考核点**:
1. `title` 是 string 类型
2. `year` 是 integer 类型（不是 string `"2017"`）
3. `rating` 是 number 类型（可以是浮点数如 8.7）
4. 必填字段 `title` 和 `year` 都存在（`rating` 可选）

**难点**: 模型容易把 `year` 返回为字符串 `"2017"` 而非整数 `2017`。这是JSON类型系统的常见错误。

---

#### 场景4: book（书籍推荐）

**考核能力**: 混合类型一致性 — 与场景3相同的schema，不同领域

**User Message**:
```
Recommend a programming book in JSON.
```

**JSON Schema**: 与场景3相同

**预期输出**:
```json
{"title": "Clean Code: A Handbook of Agile Software Craftsmanship", "year": 2008, "rating": 4.7}
```

**考核点**: 与场景3相同。如果场景3成功但场景4失败，说明模型对特定领域理解不足。

---

#### 场景5: todo（待办列表）

**考核能力**: 复合结构 — 能否返回一个**包含数组字段的对象**

**User Message**:
```
Give me a 3-item todo list in JSON.
```

**JSON Schema**:
```json
{
  "type": "object",
  "properties": {
    "items": {"type": "array"},
    "count": {"type": "integer"}
  },
  "required": ["items", "count"]
}
```

**预期输出**:
```json
{"items": ["Buy groceries", "Finish project report", "Call mom"], "count": 3}
```

**考核点**:
1. 返回的是一个**对象**（不是裸数组）
2. `items` 字段的值是一个**数组**
3. `count` 字段的值是**整数**
4. 两个必填字段都存在

**难点**: 这是三个模型差异最大的场景。MiniMax系列倾向于直接返回裸数组 `["task1", "task2", "task3"]`，而非包装在对象里。GLM-5.2则每次都能正确返回 `{"items": [...], "count": 3}` 的包装结构。

---

#### 场景6: word（词语分类）

**考核能力**: 约束遵循 — 能否遵守枚举值限制

**User Message**:
```
Classify the word 'running' in JSON.
```

**JSON Schema**:
```json
{
  "type": "object",
  "properties": {
    "word": {"type": "string"},
    "category": {
      "type": "string",
      "enum": ["noun", "verb", "adjective"]
    }
  },
  "required": ["word", "category"]
}
```

**预期输出**:
```json
{"word": "running", "category": "verb"}
```

**考核点**:
1. `word` 字段值是字符串
2. `category` 字段值必须是 `"noun"`、`"verb"` 或 `"adjective"` 之一（不能是其他值如 `"adverb"` 或 `"participle"`）
3. 两个必填字段都存在

**难点**: 模型需要同时理解"返回JSON格式"和"category值只能是三个选项之一"两个约束。这个双重约束对指令遵循能力弱 的模型是重大挑战。

---

### 1.4 考核指标定义

| 指标 | 定义 | 意义 |
|---|---|---|
| **成功率** | 20轮中返回有效JSON且通过Schema校验的比例 | 模型结构化输出的整体可靠性 |
| **Schema校验通过率** | 成功的调用中Schema校验通过的比例 | 数据正确性（零误报 = 100%意味着没有"看起来对但实际不对"的结果） |
| **平均尝试次数** | 成功调用中平均需要几次LLM调用 | 修复循环的必要性和效率 |
| **首次成功率** | 第一次调用就成功的比例（估算自平均尝试次数） | 不依赖修复循环时的"原始"能力 |

## 三、模型能力分析

### 3.1 指令遵循能力（最核心指标）

**测试什么**: 给LLM一个system prompt指令"按JSON格式返回"，看它是否服从。

| 模型 | 首次调用返回有效JSON的比例 | 评级 |
|---|---|---|
| GLM-5.2 | ~99% | **优秀** — 几乎每次都直接返回干净JSON |
| MiniMax-M2.7 | ~45% | **中等** — 约一半次数需要修复循环纠正 |
| MiniMax-M3 | ~20% | **差** — 80%的首次调用返回的是自由文本而非JSON |

**分析**: GLM-5.2在指令遵循方面表现卓越，说明它对system prompt中的格式指令理解力强。MiniMax系列（尤其M3）倾向于先输出 `<think>` 推理过程和markdown正文，忽略JSON格式要求。这个差异不是"模型不懂JSON"，而是"模型不服从格式指令"。

### 3.2 Schema复杂度适应能力

**测试什么**: 随着数据结构从简单到复杂，成功率下降幅度。

| 模型 | 简单对象 → 数组 | 简单对象 → Enum | 适应力评级 |
|---|---|---|---|
| GLM-5.2 | 100% → 100%（无下降） | 100% → 100%（无下降） | **极强** |
| MiniMax-M2.7 | 97.5% → 65%（-32pp） | 97.5% → 50%（-47pp） | **弱** |
| MiniMax-M3 | 45% → 20%（-25pp） | 45% → 40%（-5pp） | **很弱** |

**分析**: GLM-5.2对所有复杂度的schema一视同仁，无性能下降。MiniMax两个版本在数组场景都严重退化——它们倾向于直接返回裸数组 `["a","b"]` 而非包装在对象里 `{"items":["a","b"],"count":2}`。

### 3.3 数据类型准确性

**测试什么**: 返回的JSON字段值类型是否正确（string是string，integer是integer）。

| 模型 | 类型正确性 | 典型问题 |
|---|---|---|
| GLM-5.2 | 100% 正确 | 无 |
| MiniMax-M2.7 | 100% 正确（成功时） | 无 |
| MiniMax-M3 | 100% 正确（成功时） | 偶尔返回字面值如 `{"name":"string"}`（Round 13把schema类型名当成了数据值） |

**分析**: 三个模型在类型准确性上表现一致——一旦返回了有效JSON，类型几乎不会错。说明问题不在"不会用integer"，而在"不愿意返回JSON格式"。

### 3.4 修复循环配合度

**测试什么**: 当第一次返回无效内容后，引擎追加"请返回有效JSON"的反馈，模型能否在后续尝试中纠正。

| 模型 | 修复成功率 | 说明 |
|---|---|---|
| GLM-5.2 | 不适用（几乎不需要修复） | 只有2/120次触发修复，都成功了 |
| MiniMax-M2.7 | 修复后提升+37pp（45% → 82.5%） | **修复有效** — 反馈后MiniMax能理解并返回JSON |
| MiniMax-M3 | 修复后提升+20pp（20% → 40.8%） | **修复效果有限** — 即使反馈后仍经常返回非JSON |

**分析**: 修复循环对MiniMax-M2.7效果显著，但对M3效果有限。M3可能在 `<think>` 推理过程中"锁死"了输出格式，后续反馈难以纠正。

## 四、失败模式深度分析

### 4.1 失败类型分布（MiniMax-M3，71次失败）

| 失败类型 | 次数 | 占比 | 原因 |
|---|---|---|---|
| `no valid JSON found` | 44次 | 62% | 3次尝试都只返回自由文本，无任何JSON |
| `missing required fields` | 14次 | 20% | 返回了JSON但字段不完整 |
| `expected object, got array` | 13次 | 18% | 返回了裸数组而非对象 |

### 4.2 MiniMax-M3 的Round 14现象

第14轮所有6个场景全部失败——MiniMax-M3出现了**全盘崩溃**模式。查看Round 6/14/18/20都有大量全场景失败（分别为5/6、6/6、5/6、4/6失败），说明MiniMax-M3存在**间歇性的指令遗忘**。

### 4.3 MiniMax系列的共性弱点

1. **`<think>`标签污染**: MiniMax先输出长推理过程（包裹在 `<think>...</think>` 中），然后"忘记"要返回JSON格式。PAR的 `Json_extract` 能从markdown中提取JSON，但如果整段输出没有任何JSON结构，则无法恢复。

2. **数组对象混淆**: 要求返回 `{"items":[...]}` 时，MiniMax倾向直接返回 `[...]`（裸数组）。这是"理解了内容要求但没理解结构要求"的表现。

### 4.4 MiniMax-M3 Round 13的Schema回显异常

```
country-fr OK value={"name":"string","capital":"string"}
```

模型把schema定义中的类型名 `"string"` 当成了数据值。技术上通过了Schema校验（因为 `"string"` 确实是string类型），但数据内容完全错误。这种"格式正确但语义错误"的失败模式值得警惕。

## 六、结论

### 对PAR SDK的结论

PAR v0.4.8的**修复循环设计是正确的且必要的**：
- 对GLM-5.2：几乎不需要，但作为安全网存在
- 对MiniMax-M2.7：把成功率从~45%拉到82.5%，**不可或缺**
- 对MiniMax-M3：从~20%拉到40.8%，**仍然不够**——模型能力不足是根本瓶颈

### 对模型选型的建议

- **需要结构化输出的生产环境**: 首选GLM-5.2，首次成功率99%，无需额外重试成本
- **使用MiniMax的场景**: M2.7远优于M3（82.5% vs 40.8%），M3在指令遵循上反而退步了
- **MiniMax用户的调优建议**: 使用Anthropic兼容端点（`/anthropic`），设 `max_repair_attempts=5`

### 一个意外发现

MiniMax-M3（新一代）的结构化输出能力**显著弱于**MiniMax-M2.7（上一代）。这在模型升级中并不常见——通常新版本应该更强。M3可能牺牲了指令遵循能力来换取其他能力（如推理深度）。这值得MiniMax团队关注。
