# v0.4.6 Roadmap — CLI Bugfix (UTF-8 + REPL Stability)

> **Status**: Active. Created 2026-06-18.
> **Theme**: Fix the two P0 bugs that make the CLI unusable for CJK users and after tool failures.
> **Type**: Bugfix-only. Patch bump from v0.4.5.
> **Predecessor**: [v0.4.5-ROADMAP.md](v0.4.5-ROADMAP.md) (11 CLI bugfixes).

---

## Rationale

v0.4.5 shipped linenoise for REPL line editing, but two critical issues remain:

1. **CJK input broken** (PAR-qyr): linenoise treats UTF-8 bytes individually, making Chinese/Japanese/Korean input impossible (3 backspace presses per character, cursor misalignment, character corruption). This is the THIRD time this problem has been "fixed" without verification — PAR-qyr notes contain explicit lessons.

2. **REPL hangs after tool failure** (PAR-8yg): bash_confirm's `input_line stdin` (main.ml:343) blocks the Eio scheduler and corrupts linenoise's terminal state, causing subsequent inputs to get no response.

Both are P0 — the CLI is unusable for affected users.

## Work Units (4)

### WU-1 · Custom UTF-8 Line Editor (PAR-qyr) — HIGHEST RISK

> **DECISION REVISED (2026-06-18)**: Option B (revert to `input_line stdin`) was shipped instead of Option A (custom editor). The PoC gate was bypassed. PAR-qyr remains OPEN pending interactive TTY testing. See CHANGES.md for details.
>
> Rationale: Option A requires ~200 lines of new terminal management code with PoC verification that cannot be done in the current environment (no TTY). Option B relies on terminal native UTF-8 handling (canonical mode), which is battle-tested but cannot be interactively verified here either. Shipping Option B as beta with PAR-qyr OPEN is the honest path — the user tests CJK in their terminal and provides feedback.

**Complexity**: L (150-200 lines new code)
**Files**: new `bin/repl_input.ml`, `bin/main.ml`, `bin/dune`

**Problem**: linenoise doesn't handle UTF-8. Three approaches were evaluated:

| Option | Effort | CJK | Arrow Keys | Risk |
|--------|--------|-----|------------|------|
| A. Custom editor (raw mode + uutf) | L | ✅ | ✅ | Medium (new code) |
| B. Revert to input_line stdin | S | ✅ | ❌ | Low (regression on PAR-wmj) |
| C. lambda-term dependency | M | ? | ✅ | High (OCaml 5.x compat unknown) |

**Decision**: Option A (custom editor). Rationale:
- Option B regresses PAR-wmj (arrow keys) which was specifically fixed in v0.4.5
- Option C has uncertain OCaml 5.x / Eio compatibility and heavy dependency tree
- Option A uses already-installed `uutf` library, gives full control, ~150 lines

**MANDATORY PoC GATE** (per PAR-qyr lesson): Before full implementation, write a 30-line proof-of-concept that:
1. Sets raw mode via Unix.tcgetattr/tcsetattr
2. Reads a CJK character (3 UTF-8 bytes)
3. Echoes it correctly
4. Backspace deletes the full character (not one byte)
5. **Test in a real terminal** — if this doesn't work, abandon Option A and fall back to Option B

**Implementation plan** (after PoC passes):
- `bin/repl_input.ml`: raw mode setup, UTF-8 byte accumulation via uutf, escape sequence parsing (arrow keys, backspace, Ctrl-C/D), history management, terminal width awareness
- Replace `LNoise.linenoise` calls in main.ml with `Repl_input.read_line`
- Remove linenoise dependency (or keep as fallback)

**QA gate (MANDATORY)**:
- [ ] Input "你好世界" → displays correctly
- [ ] Backspace once → deletes "界" (not one byte)
- [ ] Arrow left/right → moves by character (not byte)
- [ ] Up arrow → recalls previous line from history
- [ ] Ctrl-C → first press aborts line, second within 3s exits
- [ ] Ctrl-D → exits REPL
- [ ] Mixed ASCII+CJK: "hello你好" → backspace deletes "好" not "o"

### WU-2 · Fix bash_confirm stdin conflict (PAR-8yg)

**Complexity**: S (10-20 line change)
**Files**: `bin/main.ml:337-348`, `lib/tools/bash_confirm.ml`

**Problem**: `input_line stdin` at main.ml:343 is:
1. Eio-unaware (blocks the whole domain)
2. Conflicts with linenoise/repl_input terminal state

**Fix**: Replace `input_line stdin` with the project's own line reader (LNoise.linenoise or Repl_input.read_line from WU-1):

```ocaml
(* CURRENT (broken) *)
Some (fun cmd ->
  Printf.eprintf "\n⚠ bash: %s\nProceed? [y/n]: " cmd;
  flush stderr;
  try
    let line = input_line stdin in  (* ← BLOCKS EIO + CORRUPTS TERMINAL *)
    ...
  with End_of_file -> false)

(* NEW *)
Some (fun cmd ->
  Printf.eprintf "\n⚠ bash: %s\n" cmd;
  match LNoise.linenoise "Proceed? [y/n]: " with
  | Some line -> String.trim (String.lowercase_ascii line) = "y"
                 || String.trim (String.lowercase_ascii line) = "yes"
  | None -> false)
```

**Also fix**: `lib/tools/bash_confirm.ml` fallback path (line ~37) which also uses `input_line stdin` when no `confirm_fn` is supplied.

**QA gate**:
- [ ] Trigger bash tool call → confirm prompt appears
- [ ] Type "y" → tool executes
- [ ] Type "n" → tool blocked, REPL continues normally
- [ ] After confirm, subsequent REPL inputs work correctly (the actual bug fix)
