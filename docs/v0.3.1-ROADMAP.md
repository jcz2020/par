## ✅ TOOL-1g: bash 工具

### 问题陈述

LLM 调用 shell 是**最危险的内置工具**。PAR v0.3.0 已加 6 个 read 类（read/ls/find/grep）和 2 个 write 类（write/edit）工具，但**故意没做 bash**——等 v0.3.1 单独 review。

### 核心设计

**核心理念**（来自 v0.3-ROADMAP §TOOL-1g，v0.3.1 沿用）：

> 从"裸 shell string + 黑名单"升级为"**类型化 Safe_command ADT + Policy Functor + 黑名单**"——把安全检查从运行时前移到**编译期**，体现 Q4 #1 "类型严谨"。

### 文件结构

```
lib/tools/
├── bash_safe_command.ml     (* ADT + sandboxed_path + risk 评分 *)
├── bash_policy.ml           (* POLICY Functor 接口 + 3 个预置 *)
├── bash_blacklist.ml        (* ~30 个黑名单正则 *)
└── builtin_tools.ml         (* 集成 bash 工具实现，新增 *)
```

### Safe_command ADT

```ocaml
(* lib/tools/bash_safe_command.ml *)
(* 没有 Exec_raw_shell 构造器——从类型上杜绝 shell 注入 *)

type sandboxed_path = private Path of string
val sandboxed_path_of_string : string -> (sandboxed_path, error_category) result
(** 构造时校验：拒绝 `..`、绝对敏感路径（/etc、~/.ssh 等） *)

type command =
  | Exec of {                                  (* 不接受裸 shell string！*)
      argv : string list;                       (* 必须是 argv 数组 *)
      cwd : sandboxed_path;
      env : (string * string) list;             (* 已脱敏的 env *)
      timeout : float;                          (* 秒 *)
    }
  | Pipeline of command list                    (* 已知安全命令的有限管道 *)
  | No_op                                      (* 用于测试/禁用 *)

(* 风险评分：编译期已知，运行时快速评估 *)
type risk = Low | Medium | High | Critical
val assess_risk : command -> risk
```

**关键安全约束**：
- **没有 `Exec_raw_shell`**——所有 shell 命令必须分解为 argv 数组
- 路径通过抽象类型 `sandboxed_path` 强制 CWD 相对性
- 管道仅允许连接已知安全的 `command list`
- 风险评分在 ADT 构造后即可计算（无需看运行时行为）

### Policy Functor

```ocaml
(* lib/tools/bash_policy.ml *)
module type POLICY = sig
  val name : string
  val filter : command -> (command, error_category) result
  val max_cpu_seconds : float
  val max_memory_kb : int
  val allow_network : bool
  val allow_write : bool
end

(* 预置策略——可被 Runtime.config.bash_policy 覆盖 *)
module ReadOnly : POLICY = ...        (* 任何写操作拒绝 *)
module ReadOnlyNoNet : POLICY = ...   (* + 网络拒绝 *)
module Coder : POLICY = ...           (* 默认：可写 + 可联网 + 60s timeout *)
```

**关键设计**：
- **Policy 在 `Runtime.create` 时注入**（不运行时切换）
- 不同 Agent 可绑定不同 Policy（编译期即确定）
- **Policy Functor 保证"无策略 = 无执行"**（不能"忘了配"）

### 默认 Policy 选择：🟢 Coder

**理由**（来自 v0.3 决策）：
- 与 pi 默认行为接近（写代码场景最常见）
- ReadOnly 太严，会让 LLM 抱怨"为什么我能 `find` 不能 `mkdir`"
- "无默认必须显式选"会让 80% 用户被启动配置卡住
- **Coder 配 60s timeout + 默认黑名单 + env 脱敏**已足够安全，剩余风险可由用户配置收紧

### 9 维度安全机制

| # | 机制 | 状态 | 设计 |
|---|------|------|------|
| 1 | **CWD 锁定** | ✅ 硬性不可关 | `sandboxed_path` 抽象 + 构造时校验 |
| 2 | **黑名单** | ✅ 默认 + 可关 | 黑名单在 Policy Functor 的 `filter` 实现里 |
| 3 | **白名单模式** | 🟡 可选配置 | Policy Functor 自定义实现 |
| 4 | **超时强制** | ✅ 有默认 60s | `Eio.Process.spawn` + `killpg` |
| 5 | **进程组清理** | ✅ 硬性 | `setpgid` + `killpg -SIGKILL` |
| 6 | **环境脱敏** | ✅ 有默认（**比 pi 强**） | 默认 strip `*_SECRET*` / `*_KEY*` 等 |
| 7 | **输出截断** | ✅ 有（学 pi） | 50KB byte / 2000 lines |
| 8 | **二进制清洗** | ✅ 有（学 pi） | ANSI escape 剥离 |
| 9 | **审计日志** | ✅ 走 event bus | `Bash_invoked` / `Bash_completed` 事件 |

## ✅ MCP-1: MCP stdio 客户端

### 问题陈述

2025 后 LLM 工具生态的事实标准是 [MCP](https://modelcontextprotocol.io/)。LangChain/LlamaIndex 都在接 MCP。PAR 当前没有 MCP 客户端——意味着 PAR 用户无法直接用 1000+ 现有 MCP server（filesystem / git / sqlite / github / postgres / 等）。

### 核心设计（v0.3.1 范围）

v0.3.1 = **stdio only**，实现 MCP 协议核心：
- ✅ **Tools**（LLM 调用）
- ✅ **Resources**（read-only 数据源）
- ✅ **Prompts**（预定义 prompt 模板）
- ❌ **Sampling**（server → LLM）— 推 v0.4
- ❌ **HTTP/SSE transport** — 推 v0.4

### API 设计（分层）

**高阶（config 驱动，应用开发者用）**：

```ocaml
(* runtime_config 新增字段 *)
type runtime_config += {
  ...
  mcp_servers : mcp_server_config list;     (* Runtime.create 时自动 start *)
  mcp_namespace : mcp_prefix_style;         (* 默认 Hierarchical *)
```

### Tool 命名 Pipeline

```ocaml
(* lib/mcp/mcp_naming.ml *)
type prefix_style =
  | Hierarchical
  | Flat

(* 协议 regex: ^[a-zA-Z0-9_-]{1,64}$ *)
let sanitize s =
  String.map (fun c ->
    match c with
    | ':' | '.' | '/' | '\\' -> '_'
    | c when Char.is_alphanum c || c = '_' || c = '-' -> c
    | _ -> '_'
  ) s

let mangle_tool_name ~style ~server_name ~tool_name =
  let s = sanitize server_name in
  let t = sanitize tool_name in
  let prefix = match style with
    | Hierarchical -> Printf.sprintf "mcp__%s__" s
    | Flat         -> Printf.sprintf "%s_" s
  in
  let full = prefix ^ t in
  let len = String.length full in
  if len > 60 then (
    Logs.warn (fun m -> m "MCP tool name exceeds 60 chars (will be truncated): %s" full);
    String.sub full 0 60
  ) else full
```

**安全约束**：
- ✅ `server_name` 强制用用户 config（**不信 `serverInfo.name`**，协议建议）
- ✅ 替换 `:/.\\` 为 `_`（防协议违规）
- ✅ 60 字符硬截断，50 字符 warn
- ✅ `display_title` 字段保留原始名给 UI 用

### 错误映射

```ocaml
(* lib/mcp/mcp_errors.ml *)
(* JSON-RPC 标准错误码 + MCP 自定义错误码 *)
let jsonrpc_error_to_category (code : int) (message : string) : error_category =
  match code with
  | -32700 -> Invalid_input (Printf.sprintf "MCP parse error: %s" message)
  | -32600 -> Invalid_input (Printf.sprintf "MCP invalid request: %s" message)
  | -32601 -> Invalid_input (Printf.sprintf "MCP method not found: %s" message)
  | -32602 -> Invalid_input (Printf.sprintf "MCP invalid params: %s" message)
  | -32603 -> Internal (Printf.sprintf "MCP internal error: %s" message)
  | -32000 -> Internal (Printf.sprintf "MCP server error: %s" message)
  | -32002 -> External_failure (Printf.sprintf "MCP server not initialized: %s" message)
  | -32003 -> Invalid_input (Printf.sprintf "MCP unknown resource: %s" message)
  | -32004 -> Invalid_input (Printf.sprintf "MCP unknown prompt: %s" message)
  | code when code >= -32099 && code <= -32000 ->
      External_failure (Printf.sprintf "MCP server-defined error %d: %s" code message)
  | _ -> Internal (Printf.sprintf "MCP unknown error code %d: %s" code message)
```

## 决策记录（详细依据）

### D1: v0.3.1 = bash + MCP（不含 PAR-9e1）

**触发**：2026-06-04 规划讨论。
**备选**：包含 PAR-9e1（persistence redesign）一起做。
**理由**：
- PAR-9e1 涉及 schema migration + 删 `persistence_service` 公共 API 函数 = **breaking change**
- v0.3.1 定位"post-release polish"，应**纯 additive**（不破坏现有 par.db / 用户代码）
- PAR-9e1 升为 v0.4 主题，与 MCP HTTP/SSE、MCP sampling 等其他架构变更打包
- 维持 semver 故事清晰：v0.3.x = 增量；v0.4 = 架构

### D2: bash 默认 Policy = Coder

**触发**：2026-06-04 规划讨论。
**备选**：ReadOnly 默认 / 无默认必须显式选。
**理由**：
- LLM 后端工程师（Q2 目标用户）的核心场景是"AI 写代码"，Coder 覆盖 90% 用例
- ReadOnly 太严，LLM 频繁撞墙（"为什么我能 find 不能 mkdir"）用户体验差
- 无默认会让 80% 用户卡在 `Runtime.create` 配置上
- Coder 配 60s timeout + 默认黑名单 + env 脱敏已足够安全，剩余风险由用户配置收紧

### D3: bash 风险评分只进 event bus，不进 metrics

**触发**：2026-06-04 规划讨论。
**理由**：
- v0.3.0 刚加 `Runtime.metrics` API，v0.3.1 不应反复改 metrics surface
- event bus 已经支持自定义事件（v0.3 加的 Tool_progress），风险评分挂这里扩展成本低
- v0.4+ 评估"风险评分 counter"是否值得加 metrics（要看实际 LLM 使用模式）

### D4: MCP transport = stdio only（HTTP 推 v0.4）

**触发**：2026-06-04 规划讨论。
**备选**：stdio + HTTP/SSE 同时做 / 全部 deferred。
**理由**：
- stdio 覆盖 90% 真实场景（filesystem / git / sqlite / 等本地 server）
- HTTP/SSE 估时翻倍（4-5 天 → 8-10 天），v0.3.1 控不住
- 增量交付：v0.3.1 stdio 验证协议 + API 形状，v0.4 再加 HTTP/SSE 减少 rework

### D5: MCP 生命周期 = 分层 API

**触发**：2026-06-04 规划讨论。
**备选**：Runtime 隐式管理 / 显式 register-start-stop。
**理由**：
- 符合 STRATEGY "机制 vs 策略" 原则
- 简单场景零摩擦（config 写 1 行 mcp_servers = [...]）
- 复杂场景可下沉（库作者包 start/stop 做热重载）
- 显式 `Runtime.close` 路径下自动清理防泄露

### D6: MCP 启动失败 = 可配置（默认 Fail_fast）

**触发**：2026-06-04 规划讨论。
**理由**：
- 默认 Fail_fast：避免"启动了一半 server，应用跑起来发现某些 tool 不可用"的惊喜
- 提供 Log_and_continue：本地开发 / demo 场景更友好
- 决策权交给用户，不替用户做

### D7: MCP 能力范围 = 标准 MCP（tools+resources+prompts，sampling 推 v0.4）

**触发**：2026-06-04 规划讨论（用户原话："我希望是标准 MCP 协议"）。
**理由**：
- "标准 MCP" = 协议规定的能力完整支持 = tools + resources + prompts
- sampling（server → LLM 回调）涉及双向流架构，复杂度高，v0.3.1 控不住
- v0.3.1 验证三大类 + stdio transport，sampling 跟 HTTP/SSE 一起进 v0.4

### D8: MCP tool 命名 = Hierarchical 默认 + Flat 可切

**触发**：2026-06-04 规划讨论（基于背景调研）。
**调研结论**（见 docs/v0.3.1-ROADMAP 内部 reference）：4 个生态模式
- 裸名（SDK 原生）
- 单下划线扁平（LangChain Python / LiteLLM）
- 双下划线层级（LangChain JS / Claude Code / Anthropic SDK）
- 短随机 ID（Cline，有 known bug）

**理由**：
- Claude Code / Anthropic SDK 是 Q2 目标用户最大生态
- 双下划线 `mcp__<server>__<tool>` 是协议推荐 + 主流实现一致选择
- 提供 `mcp_namespace: Hierarchical | Flat` 让用户在 token 成本 vs 兼容性之间选
- 不用短随机 ID（Cline 的 nanoid bug 模式 #8087 是反面教材）

### D9: PAR-9e1 归 v0.4（独立 minor）

**触发**：2026-06-04 规划讨论。
**理由**：
- PAR-9e1 涉及公共 API 删改（`save_events/load_events` 从 `persistence_service` 移除）+ schema migration
- v0.3.1 应**纯 additive**（不动现有 par.db / 用户代码）
- v0.4 主题已确定：PAR-9e1 + MCP HTTP/SSE + MCP sampling + bash interactive confirm
- 维持 v0.3.x = 增量 / v0.4 = 架构 的清晰 semver 故事

### D10: v0.3.1 估时 = 5-8 天

**依据**：
- TOOL-1g bash：1-2 天（v0.3 路线图估时 1-2 天，设计已就绪）
- MCP-1 stdio：3-5 天（v0.3 估时 3-5 天 + 新增 resources/prompts 实现 0.5-1 天）
- 集成测试 + 文档 + release：1 天

### D11: CHANGES.md = 独立 v0.3.1 entry

**理由**：
- v0.3.0-post 是 v0.3.0 的 hotfix/微调，**没新增 feature**
- v0.3.1 是**新 feature release**（bash + MCP），应单独 entry 让用户清楚看到新增
- 行业惯例（SemVer）：.0 → .1 之间通常有 .1 patch，但 .0+.1 = 2nd minor release
