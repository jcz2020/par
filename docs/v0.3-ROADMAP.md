## ✅ P0 战略定义（已完成）

### 1. 价值定位
**PAR = OCaml 圈的 LangChain + LangGraph**。
- MVP 友好：问答、RAG、翻译工具
- 严肃场景：8 状态机、Human-in-the-Loop、多 agent 协作
- 集成组件 + 提示词管理

### 2. 目标用户
**LLM 后端工程师**——agent 是产品一部分，重视：
- 部署性（opam + PyPI 双发布）
- 运行时稳定性、故障恢复
- 类型严谨（build-time error catching）

### 3. 差异化主张
vs LangChain (Python):
- **类型严谨**：LangChain 能崩的错，PAR 在 build 时就报
- **Eio 轻量**：同负载下 PAR 内存/CPU 仅为 Python 的 1/5
- **原生多 provider**：OpenAI / Anthropic / Ollama / Custom 零成本切换

vs pi-agent-core (TypeScript):
- **OCaml 生态唯一选择**：他们不重叠，PAR 是 OCaml 圈唯一选项

### 4. 优先级（资源分配）
1. **类型严谨**（最高）
2. **并发性能**（Eio 优势）
3. **多 provider 适配**（变现点）
4. **OCaml 生态定位**（人口天花板）

### 5. 发布策略
**opam + PyPI 双发布**：
- `par` — OCaml SDK（opam）
- `par_runtime` — Python 包（PyPI，通过 ctypes）

## UX-1: Steering / Follow-up 队列

**来源**: [pi-agent-core `steer()` / `followUp()`](https://github.com/earendil-works/pi/blob/main/packages/agent/README.md#steering-and-follow-up)

**问题**: PAR 的 `Runtime.invoke(agent_id, message)` 是单次调用返回。长跑 agent（30s+ LLM 推理 + 工具链）期间，用户无法：
- **Steer** — 中途注入新指令（如 "Stop! Do this instead"）
- **Follow-up** — 在 agent 跑完后挂后续任务（如 "Also summarize the result"）

pi-agent-core 在 TypeScript 生态验证了这个模式有效（58.9k stars，多项目使用）。

**移植设计**（适配 PAR 的 Eio 同步纤程范式）:

```ocaml
(* Runtime API 新增 *)
type steering_queue
type followup_queue

val create_steering_queue : unit -> steering_queue
val create_followup_queue : unit -> followup_queue

(* 在 invoke 期间被 Runtime 内部轮询 *)
val poll_steering : steering_queue -> (message list, error_category) result
val poll_followup : followup_queue -> (message list, error_category) result
```

**集成点** (`lib/core/engine.ml`):
- ReAct 循环每轮 tool 执行完后检查 steering 队列
- 队列非空 → 注入到对话 → 继续 LLM 调用
- 队列空 + tool 跑完 → 检查 followup 队列
- followup 非空 → 注入新消息 → 继续
- 都空 → 退出循环

**成功标准**:
- 单元测试：steer 队列非空时 ReAct 循环立即拉取并注入
- 单元测试：followup 队列在 loop 出口被检查
- 集成测试：Python 端调用 `Runtime.steer()` 真的中断正在跑的 invoke
- 集成测试：Python 端调用 `Runtime.follow_up()` 在 invoke 返回后立即触发下一轮
- 现有 198 tests 不退化

**估时**: 半天

**风险**: Eio 同步纤程与队列的并发模型需要仔细设计。建议参考 pi-agent-core 的 `PendingMessageQueue` 实现但适配 OCaml。

## UX-2: onUpdate 工具流式输出

**来源**: [pi-agent-core `AgentTool.execute` 的 `onUpdate` 回调](https://github.com/earendil-works/pi/blob/main/packages/agent/README.md#tools)

**问题**: PAR 的 13 个内置工具都是 `string -> string` 同步返回。对于长跑工具（fetch_url 抓 10MB 页面、read_webpage 解析大型 HTML、generate_password 长参数），用户看到的是"卡住 → 突然返回"。

pi-agent-core 的解决方案：tool handler 接收 `onUpdate` 回调，可以在执行过程中回传部分结果。

**移植设计**（最小侵入）:

```ocaml
(* tool_descriptor 新增字段 *)
type tool_descriptor = {
  ...
  on_update : (string -> unit) option;  (* 工具执行进度回调 *)
}

(* Handler 类型保持不变，但支持可选更新 *)
type handler_fn = tool_call -> cancellation_token -> (handler_result * (string -> unit) option)
```

**集成点**:
- `lib/providers/openai_provider.ml` 已经有 SSE 流式响应 → 流式 LLM 输出已支持
- `lib/core/engine.ml` 的 `execute_tool` 函数包装 handler 调用
- 当 handler 暴露 `on_update` 回调，engine 将其注册到 event bus，事件类型新增 `Tool_progress` variant

**成功标准**:
- 单元测试：tool handler 调用 `on_update("reading...")` 触发 event bus 事件
- 集成测试：`fetch_url` 工具在抓取大页面时每秒发一个 progress 事件
- Python 端：`Runtime.subscribe(event_type="tool_progress", callback=...)` 能收到
- 现有 198 tests 不退化

**估时**: 1 天

**不做的部分**:
- LLM 流式输出已经在 openai_provider.ml 有 SSE 实现，Python 端暂不暴露
- 不重写现有工具 handler（保持向后兼容）

## UX-3: Parallel tool execution

**来源**: [pi-agent-core `toolExecution: "parallel"` 配置](https://github.com/earendil-works/pi/blob/main/packages/agent/README.md#with-tool-calls)

**问题**: PAR 的 ReAct 循环在一次 LLM 响应中收到多个 `tool_call` 时，串行执行。LLM 经常一次吐 2-3 个 tool_call（典型场景：先 search 再 fetch）。串行执行浪费时间。

pi-agent-core 的解决方案：`executeToolCallsParallel` —— preflight（参数校验 + before_tool hook）串行做，但 tool handler 执行并发。

**移植设计**（需要重构 engine.ml）:

```ocaml
(* 新增 tool 调度器 *)
type tool_batch = {
  preflight : (tool_call * (tool_call -> tool_call option)) list;  (* 顺序：参数校验 + before_tool *)
  execution : tool_call list;  (* 并发：handler 调用 *)
  finalize : (tool_call * handler_result -> handler_result option) list;  (* 顺序：after_tool hook *)
}

val execute_tool_batch : Eio.Switch.t -> tool_batch -> (handler_result list, error_category) result
```

**集成点** (`lib/core/engine.ml`):
- ReAct 循环收齐 LLM 返回的所有 tool_call
- 第一阶段：串行 preflight（调用每个 tool 的 before_tool hook，校验参数）
- 第二阶段：用 `Eio.Fiber.all` 并发执行所有 tool handler
- 第三阶段：串行 finalize（调用 after_tool hook，按 LLM 返回顺序发射 tool_result）

**成功标准**:
- 单元测试：2 个 tool_call 并发执行，总时间 ≈ max(t1, t2) 而非 t1+t2
- 单元测试：preflight 失败时整个 batch 不执行
- 单元测试：finalize 按 LLM 源顺序发射 tool_result（不是完成顺序）
- 集成测试：3 个 fetch_url 并发执行总时间 < 串行 1/3
- 现有 198 tests 不退化

**估时**: 2 天

**风险**:
- Eio 纤程的取消语义需要重新设计（一个 tool 失败时其他 tool 的命运）
- 现有 `middleware` 的 `on_before_tool` / `on_after_tool` 钩子必须保留语义

**可配置**:
- 全局：`Runtime.config.parallel_tool_execution : bool`（默认 true）
- 单 tool：`tool_descriptor.execution_mode : [ `Sequential | `Parallel ]`（默认 Parallel）
- 串行优先：若 batch 中任一 tool 标 `Sequential`，整个 batch 降级为串行（学 pi）

## UX-4: System Prompt 设计 ✅ 方案敲定

### 选定方案：A + 行业最佳实践组合

**问题**: PAR 的 `agent_config.system_prompt` 是裸 `string`，所有 6 个文档示例都是 `"You are a helpful assistant."` 静态文本。

**选定方案（3 决策已完成）**:

1. **API 形态**：方案 A（向后兼容）
   - 新增 `system_prompt_template : system_prompt_template option` 字段
   - 保留 `system_prompt : string`（v0.2 既有）作为 Static 简化形式
   - **不破坏** v0.2 用户代码

2. **模板语法**：Mustache `{{var}}`
   - 跨语言友好（Python Jinja2、Go template、JS Handlebars 同源）
   - 学习成本低，LLM 后端工程师已熟悉
   - 语法简单（不引入控制流，降低 XSS 风险）

3. **注入变量**：系统 + tool 描述
   - `{{current_time}}` — UTC ISO 8601
   - `{{agent_id}}` — 当前 agent 标识
   - `{{runtime_id}}` — runtime 标识
   - `{{user_variables}}` — 用户提供的 key-value（Map）
   - `{{available_tools}}` — 已注册工具的 JSON 列表

4. **Provider 适配**：单一字符串，PAR 透传
   - 用户写一个 `system_prompt_template`
   - PAR 渲染为字符串
   - PAR 内部根据 provider 转换：
     - **OpenAI**: 塞入 `messages[0].role = "system"`
     - **Anthropic**: 塞入顶层 `system` 字段
     - **Ollama**: 塞入 `messages[0].role = "system"`
     - **Custom**: 用户自己决定
   - **不暴露** system vs developer role 区别（学 LangChain / pi-agent-core / Vercel AI SDK）

### API 形态

```ocaml
(* lib/core/types.ml 新增 *)
type system_prompt_template = {
  template : string;          (* Mustache 模板 *)
  variables : string list;    (* 允许的变量名白名单（type safety） *)
  required : string list;      (* 必填变量（缺则报错） *)
}
[@@deriving yojson]

(* agent_config 新增字段（v0.2 既有 system_prompt 保持不变） *)
type agent_config = {
  id : string;
  system_prompt : string;                          (* 静态文本，v0.2 兼容 *)
  system_prompt_template : system_prompt_template option;  (* v0.3 新增，optional *)
  model : model_config;
  ...
}
```

### 渲染流程

```
user template + variables dict
     │
     ▼
┌────────────────────────────────────────┐
│ Mustache.render(template, vars)       │
│   - 替换 {{var}}                        │
│   - 验证 vars ⊆ whitelist (类型严谨)   │
│   - 必填检查 (required 都存在)         │
└────────────────────────────────────────┘
     │
     ▼
Rendered string
     │
     ▼
┌────────────────────────────────────────┐
│ provider adapter (按 model.provider)  │
│   - Openai  → messages[0].system      │
│   - Anthropic → top-level system       │
│   - Ollama  → messages[0].system       │
│   - Custom  → custom_fn user defined   │
└────────────────────────────────────────┘
```

### 行业对照（验证方案选择）

| 框架 | 暴露给用户 | 自动转换 | 适配模式 |

## 依赖关系

```
FFI-1 (基础) ──→ FFI-2 (依赖 FFI-1 真实现)
UX-1 (独立) ──→ 可与 FFI 并行
UX-2 (依赖 event bus 扩展) ──→ 可与 UX-1 并行
UX-3 (依赖 engine.ml 重构) ──→ 需独立串行（避免合并冲突）
UX-4 (独立) ──→ 可与所有其他并行
```

**建议执行顺序**（按 Q4 优先级 1:类型严谨 > 2:并发 > 3:多 provider）:

**Wave 1**（并行）:
- **UX-4**（System Prompt）— 直接提升类型严谨 + 多 provider
- **FFI-1**（Python SDK 完整化）— 必做基础

**Wave 2**（并行）:
- **FFI-2**（依赖 FFI-1）
- **UX-1**（steering/followup）— 提升并发交互价值

**Wave 3**（并行）:
- **UX-2**（onUpdate）— 提升并发性能
- **UX-3**（parallel tool）— 提升并发性能

**原则**：类型严谨（UX-4）排第一，Q4 优先级 1 直接要求。
