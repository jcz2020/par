# v0.5 Roadmap

> **Goal**: Ship platform-true wheel for ARM64 macOS, in addition to the existing x86_64 Linux manylinux_2_28 wheel.
> **Rationale**: Current `pip install par-runtime` crashes on Apple Silicon Macs (loads x86_64 ELF via ctypes). v0.5.0 fixes that without forcing source-build for Mac users.
> **Released**: 2026-06-21

---

### Deferred to v0.5.1+ (NOT in v0.5.0)

**ARM64 Linux wheel** (`manylinux_2_28_aarch64`): attempted via 3 paths, all blocked:
1. `ubuntu-22.04-arm64` / `ubuntu-24.04-arm64` GH-hosted runners — queue 45min+ on free tier, never dispatched
2. qemu-binfmt on `ubuntu-22.04` host with `quay.io/pypa/manylinux_2_28_aarch64:latest` container — container crashes on start
3. OCaml cross-compile toolchain — Oracle-verified as not production-viable for ctypes binding linking system GMP/sqlite3

**User decision 2026-06-21**: skip ARM64 Linux for v0.5.0, defer to v0.5.1+. Tracked in bd issue `PAR-5oq` (P3). Recovery signals documented in `docs/STRATEGY.md`.

### Why no universal2 (Intel Mac + ARM Mac fat binary)

`macos-13` (Intel) runner was permanently abandoned 2026-06-19 (`ci.yml` L16) — free-tier Intel runners queue 24h+ then hit max-execution-time. True `universal2` requires BOTH `macos-13` + `macos-15` slices; not achievable without paid GH Actions minutes.

Trade-off accepted:
- Apple Silicon Macs (the 2026 market) get a native wheel
- Intel Macs (<5% of PyPI 2026 traffic) fall back to `par-runtime==0.4.13` or source build
- Defer Intel Mac native wheel to v0.6+ (when/if user revisits the macos-13 decision or moves to a paid tier)

### Changes from v0.4.13

**`pypi-publish.yml`** — refactored from single hardcoded job into a 2-job matrix:
- Job `build-wheel (linux-x86_64)` — preserves v0.4.13 behavior (manylinux_2_28_x86_64 container + auditwheel)
- Job `build-wheel (macos-arm64)` — NEW: `macos-15` runner, `delocate-wheel` for dylib bundling, `wheel tags --platform-tag macosx_11_0_arm64` to set the platform tag (ctypes .so gets `py3-none-any` by default from setuptools; filename-only rename fails PyPI 400 — internal WHEEL Tag field must match)
- `pypi-upload` job downloads both artifacts and publishes them in one PyPI upload call

**`release-acceptance.yml`**:
- Removed `WHEEL_COUNT -ne 1` guard — release now ships 2 wheels
- Split into `accept-linux-x64` (3-container matrix: debian:12, ubuntu:22.04, ubuntu:24.04) + `accept-macos-arm64` (native macos-15 runner)
- Added missing `actions/checkout@v4` step (macOS job was missing `scripts/release-acceptance-test.py`)

### Out of scope for v0.5.0 (deferred)

- **ARM64 Linux wheel**: deferred to v0.5.1+ (bd PAR-5oq)
- **Intel Mac native wheel**: requires `macos-13` runner (rejected) OR paid GH Actions (deferred)
- **musllinux wheels** (Alpine, Clear Linux): not requested; manylinux covers ≥95% of Linux PyPI installs
- **Windows wheel**: ctypes on Windows requires MSVC; out of scope, source-build only on Windows
- **`-manylinux_2_17` (RHEL 7 / CentOS 7)**: user explicitly rejected 2026-06-21 ("老的Linux我觉得完全可以不必兼容")
- **RAG Foundation** (originally planned for v0.5.0 per `v0.5.0-ROADMAP.md`): superseded by multi-arch wheel work; RAG moved to v0.5.1+
