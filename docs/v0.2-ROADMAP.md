## 概述

v0.1.0 已完成 171 个测试（163 OCaml + 8 Python），发布论文，opam 就绪。v0.2.0 聚焦三类工作：

| 优先级 | 主题 | 核心问题 |
|--------|------|----------|
| P0 | 正确性 | stub 实现永远不触发超时，runtime 会在畸形 JSON 上崩溃 |
| P1 | 质量 | 硬编码配置值散落 7 处，`ignore()` 掩盖错误 |
| P2 | 功能 | DESIGN.md 中规划但未实现的能力 |

## P0 — 正确性（必须修复，否则 runtime 不可信）

### P0-1: `with_timeout` 忽略 timeout 参数

**问题**: `cancellation.ml:16` 的 `with_timeout _seconds` 用下划线丢弃参数，超时永远不会触发。

**文件**: `lib/core/cancellation.ml:16`

**修复方案**: 实现真正的超时机制。使用 `Eio.Fiber.with_timeout` 或在函数内用 `Eio.Clock.now` 计算截止时间。

**成功标准**:
- 写入测试：给定 100ms 超时，200ms 的操作返回 `Error Timeout`
- 现有 163 个 OCaml 测试全部通过

---

### P0-2: `Human_approval` 忽略 timeout 参数

**问题**: `workflow_engine.ml:98` 的 `Human_approval { timeout = _; ... }` 丢弃 timeout 参数，导致等待人工审批时永远挂起。

**文件**: `lib/core/workflow_engine.ml:98`

**成功标准**:
- 写入测试：带 1s 超时的人工审批步骤在无审批时返回 `Error Timeout`
- 集成测试覆盖 `Human_approval` 超时路径

---

### P0-3: yojson 转换器用 `failwith` 崩溃 runtime

**问题**: `types.ml:689-706` 等多处 `task_completion_of_yojson` 等函数在 JSON 格式错误时调用 `failwith`，直接崩溃整个 runtime 进程。

**文件**: `lib/core/types.ml:689,695,698,700,703,706`

**修复方案**: 将 `failwith` 替换为返回 `Result.Error` 类型的 `parse_yojson` 函数，保持错误信息返回而不是抛异常。

**成功标准**:
- 向 `types.ml` 追加 yojson 转换测试，用畸形 JSON 验证不崩溃
- 所有 yojson 解析函数返回 `Result` 类型，不再抛出异常

---

### P0-4: 多处 `ignore()` 掩盖错误状态

**问题**: cleanup/rollback 代码中用 `ignore()` 丢弃返回值，失败静默。运行时完全不知道持久化层出了问题。

**文件**:
- `lib/persistence/sqlite_persistence.ml:59,61,231,232`
- `lib/persistence/postgres_persistence.ml:54,256`
- `lib/ffi/par_capi.ml:40,71,81`

**修复方案**: 替换为结构化日志记录。cleanup 错误不阻断主流程，但必须可观测。

**成功标准**:
- cleanup 路径中所有 `ignore()` 替换为 `Logs.err` 或等价日志
- 测试验证：故意的 cleanup 失败产生可观测的错误日志

---

## P1 — 质量（提升可配置性和健壮性）

### P1-1: 表达式求值器硬编码资源限制

**问题**: `expression.ml:5-6` 的 `max_depth=10` 和 `max_node_visits=1000` 写死在代码里，无法按调用场景调整。

**文件**: `lib/core/expression.ml:5-6`

**修复方案**: 通过 `runtime_config` 传入或 `eval_context` 携带限制参数。

**成功标准**:
- 能在创建 runtime 时覆盖这两个值
- 测试覆盖：超过限制时返回 `Error (Internal "Expression evaluator: max ... exceeded")`

---

### P1-2: runtime 配置硬编码

**问题**: `runtime.ml:20-39` 的 `buffer_capacity`、`max_concurrent_tasks`、`drain_timeout`、`default_max_tokens` 均硬编码。

**文件**: `lib/core/runtime.ml:20-39`

**成功标准**:
- `runtime_config` 包含所有运行时可配置项
- `Runtime.create` 能接受自定义值而非依赖默认值
- 测试：用非默认配置创建 runtime，验证行为符合配置

---

### P1-3: 中间件硬编码参数

**问题**: `retry.ml:7-10` 的 `max_attempts=3`/`base_delay=2.0`/`max_delay=30.0` 和 `rate_limit.ml:7` 的 `max_requests=60`/`window=60.0` 不可配置。

**文件**:
- `lib/middleware/retry.ml:7-10`
- `lib/middleware/rate_limit.ml:7`

**修复方案**: 通过 `runtime_config.middleware` 或单独的配置函数传入。

**成功标准**:
- `retry` 和 `rate_limit` 构造函数接受非默认参数
- 测试：配置 `max_attempts=1` 时验证重试行为

---

### P1-4: `validation` 中间件错误不向上传播

**问题**: `validation.ml:21,41` 在 strict=false 时只记录 `Logs.warn`，错误被吞掉，调用者不知道发生了数据修复。

**文件**: `lib/middleware/validation.ml:21,41`

**修复方案**: 无论 strict 模式如何，都返回错误结果；strict 控制是否接受修复后的数据。

**成功标准**:
- strict=true：无效输入直接返回 Error
- strict=false：无效输入被静默修复但仍记录 warn
- 测试覆盖两种模式

---

### P1-5: 重试中间件并发安全性审查

**问题**: `retry.ml:53` 的 `let attempt = ref 0` 在每次 `retry ()` 调用时创建，单实例内并发调用共享同一 ref。需验证多 fiber 同时触发 retry 时计数器竞态是否安全。

**文件**: `lib/middleware/retry.ml:53`

**修复方案**: 若竞态存在，使用 `Eio.Ref` 或 per-invocation 状态传递。

**成功标准**:
- 审查并发安全性，确认或否定竞态问题
- 若有问题：并发调用同一 agent 两次，retry 计数独立
- 若无问题：在代码中添加注释说明安全性保证

---

## P2 — 功能（按依赖顺序排列）

### P2-1: 为 `builtin_tools` 补充单元测试

**问题**: 13 个内置工具没有独立单元测试，只通过集成测试间接覆盖。单个工具行为无法独立验证。

**文件**: `lib/tools/` 下 13 个工具文件

**成功标准**:
- 每个内置工具有至少 3 个独立的快速单元测试
- 测试覆盖：正常输入、边界输入、错误输入
- 163 个 OCaml 测试总数增加至 190+

---

### P2-2: `http_client.ml` 共享 HTTP 层

**问题**: OpenAI 和 Anthropic provider 各有约 80% 重复的 HTTP 代码（URL 解析、chunked encoding、TLS 配置）。

**文件**: `lib/providers/openai_provider.ml`, `lib/providers/anthropic_provider.ml`

**修复方案**: 提取公共 HTTP 逻辑到 `lib/providers/http_client.ml`，两个 provider 调用共享层。

**成功标准**:
- 两个 provider 的网络请求通过同一个 http_client 实例发出
- 减少约 200 行重复代码
- 现有集成测试全部通过（网络请求行为不变）

---

### P2-3: `sanitize_tool_output` — 提示词注入防御

**问题**: 外部工具返回的内容直接拼入 prompt，存在提示词注入风险（恶意工具输出植入指令）。

**文件**: DESIGN.md 规划，`lib/middleware/` 待实现

**修复方案**: 实现 `sanitize_tool_output` 中间件，过滤工具输出中的潜在指令模式（如 `You are`、`Ignore previous`、`System:`）。

**成功标准**:
- 中间件存在且可挂载到 tool 输出链
- 测试：包含注入模式的工具输出被正确过滤或标记
- 集成测试验证不过滤正常内容

---

### P2-4: `postgresql` 改为可选依赖

**问题**: PostgreSQL 持久化后端是必选依赖，未安装 PG 时 dune 构建失败。

**文件**: `dune`, `lib/persistence/postgres_persistence.ml`

**修复方案**: 使用 dune 的 `optional` 依赖声明，提供一个 `Noop_persistence` fallback。

**成功标准**:
- 无 PostgreSQL 安装时 `dune build` 成功
- PG 不可用时 runtime 使用 `noop_persistence` 不报错
- PG 可用时正常加载

---

### P2-5: `par_register_tool` FFI 返回值修复

**问题**: `par_capi.ml:81` 的 `par_register_tool` stub 处理器始终返回 0（成功），无法反映真实注册结果。

**文件**: `lib/ffi/par_capi.ml:75-87`

**修复方案**: 实现真正的工具注册 handler，从 JSON schema 解析 `input_schema`，调用 `Par.Runtime.register_tool`。

**成功标准**:
- Python 端 `par_runtime.register_tool()` 正确返回注册状态
- 错误的 schema JSON 返回 -1
