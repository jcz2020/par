    v0.4 Roadmap — Event Persistence + Architecture Upgrade¶

    Status: Draft v2. Updated 2026-06-09 after Oracle review of 6-framework agent conversation research. Theme: Make the persistence layer actually work end-to-end, add typed agent handoff, expand MCP transport, and unblock Python tool authors. Strategic change: STRATEGY.md §3.2 "不做多 agent 协作" restriction lifted 2026-06-09. Handoff (single-chain A→B via typed ADT) now in scope.

    Background¶

    v0.3 shipped a working SDK (20 tools, 7 middleware, MCP stdio, workflow engine, streaming CLI, 871 tests). The persistence layer is the biggest gap: SQLite, PostgreSQL, and Noop backends exist, the interfaces are exported, but the Runtime engine never calls save_events. SDK users see a misleading API.

    v0.4 closes this gap and adds typed agent handoff, structured output validation, MCP transport upgrades, Python callback support, and several smaller items deferred from v0.3.1.

    Research Basis¶

    7 mainstream agent frameworks were researched and Oracle-reviewed in 2026-06-09:

    Framework Key Takeaway for PAR
    LangGraph 3-table persistence (checkpoints + blobs + writes); Checkpointer per thread_id
    OpenAI Agents SDK Handoff = synthetic transfer_to_<agent> tool; RunContextWrapper DI (never sent to LLM)
    AutoGen BaseChatMessage/BaseAgentEvent; save_state/restore_state serialization
    Google ADK EventActions (state_delta, transfer_to_agent); 3-level state prefix (app:/user:/session:)
    PydanticAI Structured output = synthetic output tool; RunContext for deps; message_history passthrough
    Anthropic append-only content blocks; Session Runner for disconnect recovery (not needed for PAR)
    CrewAI Task.context for prior output passthrough; ContextualMemory; unified memory in v1.14

    Cross-cutting consensus: flat append-only message array; session_id/thread_id isolation; context window = sliding window + large content offload; handoff via synthetic tool (Python) or typed event (OCaml).

    Sub-Version Plan¶

    v0.4 is broken into 5 sub-versions, each independently shippable with a clean test suite.

    v0.4.0 ── Phase 1: Python callback + Bash confirm + Structured output middleware
v0.4.1 ── Phase 2: Event persistence + Session_id + CLI history/stats
v0.4.2 ── Phase 3: Typed Agent Handoff + EventActions schema column + par replay
v0.4.3 ── Phase 4: MCP HTTP/SSE + MCP Sampling + evaluations
v0.4.4 ── Phase 5: CLI upgrade — handoff support, session management, output formatting

    v0.4.1 — Event Persistence + Session Scoping (Phase 2)¶

    Goal: Make the persistence layer actually work end-to-end. Add session_id to events. CLI commands for history/stats. Estimated: 5–6 days (Oracle-reviewed, up from original 5–10 day estimate but scoped tighter).

    2.1 Persistence as Event Bus Tail Consumer¶

    Source: Original P2 bead, deferred from v0.3.1. Oracle review: don't thread save_events_fn into engine — subscribe persistence as event-bus consumer instead.

    Current state: save_events_fn / load_events_fn interfaces defined in PERSISTENCE_SERVICE module type. Three implementations compile. Runtime.create sets save_events_fn = (fun _events -> Ok ()) (noop). Engine never calls it.

    Architecture decision: Persistence subscribes to the event bus as a tail consumer. Engine stays pure (no I/O). Event bus fans out to both in-memory subscribers and the persistence writer.

    Engine → on_tool_event callback → Event_bus.publish → in-memory subscribers
                                                  → Persistence writer fiber (batched)

    Scope: 1. Subscribe persistence to event bus: On Runtime.create, register the persistence backend as a subscriber. Events flow through the existing publish path; no engine changes needed. 2. Async buffered writer: Persistence writes happen in a background fiber via Eio.Fiber.fork_promise. Bounded buffer (configurable, default 1000 events). Flush on Runtime.close. 3. Batch writes: Micro-batch with configurable flush interval (default 50ms). Reduces SQLite lock contention under high event volume. 4. DLQ interaction: When persistence buffer is full, overflow events go to the existing DLQ. No event loss — worst case is delayed persistence. 5. Retention: Configurable TTL (default 7 days). Prune expired events on Runtime.create. 6. Schema: Reuse existing events table. Add session_id column (see 2.2). Add actions_json column (reserved for v0.4.2, nullable).

    Estimated: 3–4 days.

    Success Criteria: - [ ] Runtime.create ~config:{ persistence = \Sqlite "par.db"; ... }persists every event to the database - [ ] Persistence writes are async — agent loop is never blocked by SQLite/PG writes - [ ]Runtime.close` flushes buffered events before returning - [ ] DLQ catches overflow when buffer is full - [ ] Existing 871 OCaml tests + 16 Python tests pass (zero regressions) - [ ] New tests: persistence integration, buffer overflow, retention pruning

    2.2 Session_id Wiring¶

    Source: Oracle review: Session_id type exists but is unused on events. Without session scoping, persisted events are an unqueryable firehose.

    Current state: Session_id.t defined at types.ml:40-49. Created once at runtime init as runtime_id. Not on event_metadata. Not indexed by persistence. extract_task_id returns "" for 14 of 28 event variants.

    Scope: 1. Add session_id : string to event_metadata (types.ml:372-377) 2. Event_bus.publish threads session_id — capture from bus config (one bus per runtime instance) 3. Session granularity: one Runtime.Invoke call = one session. Each invoke generates a fresh session_id. This enables "show me all events for this conversation." 4. All three persistence implementations: add session_id column, index, update queries 5. extract_session_id companion to extract_task_id — reads from envelope metadata, not payload

    Estimated: 2–3 days (touching types, event_bus, 3 persistence impls, schema).

    Success Criteria: - [ ] Every event envelope carries session_id - [ ] load_events can query by session_id (new overload or parameter) - [ ] Session granularity = one Runtime.invoke call - [ ] Existing tests pass

    3.1 Typed Agent Handoff¶

    Source: OpenAI Agents SDK, AutoGen, Google ADK research (2026-06-15 cross-framework study). STRATEGY.md §3.2 restriction lifted 2026-06-09.

    Current state: Runtime.register_agent stores agents in a hashtable. Engine.run_agent receives a single agent_config (immutable binding). No agent-switching primitive exists. The workflow engine has agent_resolver pattern (workflow_engine.ml:35, plumbed at runtime.ml:465,483,561) that can be reused.

    Architecture decision: OCaml typed ADT, NOT JSON synthetic tool (preserves STRATEGY Q4 priority #1: type rigor). Strictly stronger than OpenAI's JSON-encoded function_call_output, LangGraph's metadata-encoded ToolMessage.response_metadata, and ADK's Optional[str] flag — compile-time exhaustiveness checking on handoff handling is impossible in all three reference frameworks.

    Scope: 1. Add ?agent_resolver to Engine.run_agent: (string -> agent_config option) closure, plumbed from Runtime.invoke via (fun aid -> htbl_get rt.agents aid). Reuses pattern from workflow_engine.ml:35. Also add ?enable_handoff:bool (default false) for backward compatibility. 2. New handler_result variant (revised 2026-06-15 per Oracle review): 

    | Handoff of {
    target_agent_id : string;
    carry_context : bool;       (* true = preserve conversation history (default) *)
    task : string option;        (* when carry_context=false, becomes target's initial user message *)
  }
    The task field is required when carry_context=false (else Invalid_input error). When carry_context=true, task is optional and treated as an additional instruction appended to the conversation. 3. Engine handoff logic (revised 2026-06-15 per Oracle review — see Design Decisions Log below): - Refactor let rec loop conv iterations → let rec loop ~agent ~global_max conv iterations (thread agent + global_max through recursion; 5 recursive call sites at engine.ml:217,237,334,339,343) - Before the tool-result fold at engine.ml:330-332, partition results into (non_handoff, handoffs) via List.partition - Fold non_handoff results into conv first (preserve their tool outputs) - If handoffs = []: normal recursion path - If handoffs has exactly 1 entry AND enable_handoff=true: resolve target via agent_resolver, fire Agent_handoff event, recurse with ~agent:target_agent ~global_max - If handoffs has >1 entry: Result.Error (Invalid_input "Multiple handoffs in one tool batch", conv) (ambiguous, fail loud — matches OpenAI Agents SDK turn_resolution.py:426) - If handoffs has 1 entry AND enable_handoff=false: Result.Error (Invalid_input "Tool returned Handoff but enable_handoff=false", conv) - On resolver miss (agent_resolver target_agent_id = None): Result.Error (Invalid_input "Handoff target not found: <id>", conv) — matches workflow_engine.ml:88 convention - Iteration budget: use running max-of-chain global_max = max global_max agent.max_iterations (see Decision 4 below for rationale) - Middleware chain: implicit swap — target agent's full chain takes over on next LLM call (see Decision 3) - System_prompt: silently replaced — target agent's system_prompt is used on next LLM call, source agent's is dropped (matches OpenAI Agents SDK default; see Decision 1) 4. New event type: Agent_handoff of { from_agent : string; to_agent : string; task_id : Task_id.t } for observability. The task_id MUST link to the triggering tool call (use the tool_call.id from the handoff result, not a fresh Task_id.create ()) for par replay traceability. 5. API surface: - Runtime.invoke gains ?enable_handoff:bool (default false) - When true, passes ~agent_resolver:(fun aid -> htbl_get rt.agents aid) + ~enable_handoff:true to Engine.run_agent at runtime.ml:358-368 6. Exhaustiveness updates (11 match sites on handler_result): - engine.ml:90-105 (execute_tool output-schema check): | Handoff _ -> result (passthrough) - engine.ml:107-109 (progress log): | Handoff _ -> progress "Tool handoff" - engine.ml:166-178 (add_tool_result_message): | Handoff _ -> "<handoff>" (unreachable in normal path due to partition, but type-exhaustive) - engine.ml:237 (retry decision): | Handoff _ -> Result.Error (Internal "Handoff reached retry path", conv) (defensive) - engine.ml:310-318 (event firing): handled by partition logic above - middleware/logging.ml:22, pii_mask.ml:73, output_validation.ml:20,34, sanitize_tool_output.ml:73: passthrough arms - workflow_engine.ml:102: | Handoff _ -> Result.Error (Invalid_input "Handoff not supported in workflow Agent_call", ...) (workflows don't hand off) 7. Persistence: extract_task_id at persistence_common.ml:34 gains | Agent_handoff { task_id; _ } -> Task_id.to_string task_id as 15th task-bearing arm.

    Estimated: 5.5–7.5 days (revised 2026-06-15; +0.5 day for running-max threading, task field yojson/FFI ripple, and partition-before-handoff logic).

    Design Decisions Log (Oracle-reviewed 2026-06-15)¶

    The original 4 "hard questions" (lines 215-219 of pre-revision ROADMAP) are now RESOLVED. Each was researched across OpenAI Agents SDK / LangGraph / AutoGen / Google ADK and Oracle-reviewed:

    # Question Decision Industry basis
    D1 How does handoff appear in message history (mixed system_prompt framing)? Source system_prompt replaced with target agent's system_prompt + carry_context: bool field. When carry_context=true (default): preserve conversation history (user/assistant/tool messages), strip source agent's System message, prepend target agent's System message computed via Template.effective_system_prompt. When false: fresh conversation with target's System message + task field as the User message. No HandoffMessage marker inserted into conversation (unlike AutoGen) — the Agent_handoff event provides observability via event bus instead. OpenAI Agents SDK default + Google ADK consensus (silent replacement). Verified via OpenAI Agents SDK handoffs doc: "the new agent takes over the conversation, and gets to see the entire previous conversation history" — instructions is per-Agent, injected at call time, never stored in items. PAR differs architecturally: system_prompt IS stored as conversation message[0], so we actively replace it rather than rely on per-call injection. nest_handoff_history is opt-in beta, disabled by default. AutoGen's HandoffMessage is cleanest but adds a message type PAR doesn't need.
    D2 Tool_call_ids when target agent lacks tools? Default: carry as-is. OpenAI/Anthropic APIs tolerate historical tool_calls not matching current tool set (they're transcript items, not contract requirements). No filter implementation in v0.4.2. OpenAI Agents SDK default carries history flat; remove_all_tools is an opt-in filter. gpt-5 reasoning-chain breakage (issues #985, #2020) makes filtering a foot-gun — defer to v0.4.3 with proper reasoning-item handling.
    D3 Middleware chain switch mid-conversation? Implicit full swap + per-agent state isolation. Target agent's full middleware chain takes over on next LLM call. Stateful middleware (Retry counter, Rate_limit token bucket) is FRESH for target agent — does NOT inherit source agent's retry attempts or rate budget. Self-handoff retry-bypass threat is negligible (agents are dev-configured, not adversarial) and is backstopped by D4 iteration budget. Divergence from OpenAI: OpenAI scopes input-guardrails to first agent, output-guardrails to last agent. PAR's full swap is more aggressive but simpler and safer (no silent Validation bypass). All 4 frameworks do implicit swap. Per-agent state isolation matches AutoGen fresh-budget semantics.
    D4 max_iterations budget semantics? Inherit iterations + check against running max-of-chain. Track global_max = max global_max agent.max_iterations as agents swap. Check is iterations >= global_max. This replaces the original "reset counter" plan (line 208 of pre-revision ROADMAP) which would have allowed A→B→A→B infinite loops. OpenAI Agents SDK + LangGraph + Google ADK all INHERIT (single global counter). AutoGen is lone dissenter (per-agent fresh, but has separate group-chat budget). PAR has no group-chat layer, so invoke-level budget is the only safety net. Running max-of-chain (instead of plain inherit) avoids the instant-exit surprise: A(max=10) hands off at iter=5 to B(max=3) → check 5 >= max(10,3)=10 → B gets 5 more iterations, instead of instant-exit under plain "check against target's max".

    Why typed ADT is strictly stronger than the alternatives (strategic differentiator):

    Framework Handoff signal representation Compile-time exhaustiveness?
    OpenAI Agents SDK JSON-encoded function_call_output ❌
    LangGraph ToolMessage.response_metadata[METADATA_KEY_HANDOFF_DESTINATION] (string-keyed dict) ❌
    Google ADK EventActions.transfer_to_agent : Optional[str] ❌
    AutoGen HandoffMessage (typed) — closest to PAR ⚠️ Partial (Python mypy only)
    PAR Handoff of { target_agent_id; carry_context; task } ADT constructor ✅ OCaml exhaustive match

    The cost is one extra type the user must learn; the payoff is compile-time exhaustiveness checking on handoff handling, which is impossible in any of the four reference frameworks.

    Common pitfalls defended against (from cross-framework bug research)¶

    1. Multiple handoffs in one tool batch — OpenAI silently ignores the 2nd+; PAR errors loudly (Invalid_input "Multiple handoffs in one tool batch").
    2. gpt-5 reasoning-chain breakage when filtering history — PAR doesn't filter by default (D2), sidestepping this entirely.
    3. ADK transfer_to_agent from after_model_callback silently ignored (issue #5125) — PAR's handoff works from any tool result uniformly, no lifecycle-hook restrictions.
    4. LangGraph full-history handoff "confuses" target agent (per official docs) — PAR's carry_context=false escape hatch gives users the option to reset.
    5. Tool-call/Tool-message pairing invariant — when handoff is the sole result (after partition), no orphaned tool_call_ids reach the LLM API.
    6. Per-agent identity in tracing — Agent_handoff event records from_agent + to_agent explicitly for observability; downstream consumers (CLI, metrics) never lose track of the active agent.

    Success Criteria: - [ ] Agent A can hand off to Agent B mid-conversation (happy path: carry_context=true, task=None) - [ ] Agent A can delegate fresh task to Agent B (carry_context=false, task=Some "...") - [ ] carry_context=false without task returns Invalid_input error - [ ] Conversation history carries across handoff when carry_context=true - [ ] Conversation is reset when carry_context=false - [ ] Source agent's system_prompt is silently dropped (target's takes over) - [ ] Iteration budget uses running max-of-chain (A max=10 → B max=3 at iter=5 → B runs until iter=10, not instant-exit) - [ ] Handoff event Agent_handoff { from_agent; to_agent; task_id } appears in event bus - [ ] task_id in event links to triggering tool call (not fresh) - [ ] Multiple handoffs in one tool batch → Invalid_input error - [ ] Resolver miss → Invalid_input "Handoff target not found" error - [ ] ?enable_handoff=false (default) preserves exact current behavior (tool returning Handoff → Invalid_input error) - [ ] Middleware state (Retry counter, Rate_limit bucket) is per-agent, not inherited across handoff - [ ] Type-safe: handoff signal is an ADT constructor, not a JSON convention - [ ] All 11 exhaustive match sites on handler_result updated - [ ] extract_task_id handles Agent_handoff variant - [ ] 10+ new tests covering: happy path, carry_context variants, resolver miss, multiple-handoff error, enable_handoff=false error, middleware isolation, budget running-max, event emission

    5.1 Multi-Agent REPL with Handoff Display¶

    Source: v0.4.2 ships typed agent handoff in the SDK. The CLI currently runs a single default-agent and has no concept of agent switching.

    Current state: bin/main.ml registers one agent (default-agent, line 282-287), the REPL invokes it in a loop (line 402). The make_tool_event_callback (line 310-332) prints → tool_name ✓ (Nms) for tool events but has no handler for Agent_handoff events.

    Scope:

    1. Config-driven multi-agent: Extend par_config.ml to accept an agents array in ~/.par/config.json. Each entry has id, system_prompt, model (optional, inherits default), tools (optional subset), max_iterations (optional). Example: 

      {
  "agents": [
    { "id": "researcher", "system_prompt": "You are a research assistant..." },
    { "id": "writer", "system_prompt": "You are a writing assistant...", "model": "claude-sonnet-4-20250514" }
  ]
}
      If no agents array: current behavior (single default-agent). Backward compatible.

    2. Agent registration at startup: setup_runtime reads the agents array, registers each one with Runtime.register_agent. First agent in array = default.

    3. Handoff display in REPL: When Agent_handoff { from_agent; to_agent } event fires, print to stderr: 

      ↪ researcher → writer
      (Matches existing → tool ✓ (Nms) style — tool events on stderr, stdout clean.)

    4. Handoff indicator in prompt: After a handoff, the par> prompt shows the active agent: 

      par [researcher]> hello
...
↪ researcher → writer
par [writer]> continue

    5. /agents REPL command: List registered agents and their status. 

      /agents
  researcher  (active, 3 turns)
  writer      (idle)

    6. /switch <agent_id> REPL command: Explicitly switch agents (user-initiated, not LLM-initiated handoff).

    Estimated: 2 days.

    Success Criteria: - [ ] ~/.par/config.json with agents array registers multiple agents - [ ] Backward compatible: no agents array = single default-agent (current behavior) - [ ] Handoff events show ↪ from → to in stderr - [ ] Prompt shows active agent: par [agent_id]> - [ ] /agents lists registered agents - [ ] /switch <id> changes active agent - [ ] 3+ new tests

    5.3 Bash Confirmation Prompt¶

    Source: v0.4.0 §1.2 ships bash interactive confirmation in the SDK. The CLI needs the UI.

    Current state: bin/main.ml:298-302 installs a tool-call hook that prints tool names to stderr ([tool_name]). No Confirm verdict handling.

    Scope:

    1. Hook returns Confirm: The existing tool_call_hook (line 298) returns Hook.Allow. Change to check the bash command against a configurable confirmation list. When matched, return Hook.Confirm.

    2. REPL prompt: When engine receives Confirm, pause the ReAct loop, print the command, wait for [y/n]: 

      par [researcher]> delete the temp files

⚠ bash: rm -rf /tmp/test
Proceed? [y/n]: y
→ bash ✓ (123ms)

Done. The files have been removed.

    3. Configurable rules: In ~/.par/config.json: 

      {
  "bash_confirm": {
    "patterns": ["rm -rf*", "dd *", "mkfs*"],
    "default": "allow"
  }
}
      default can be allow (confirm only matched patterns) or confirm (confirm everything).

    4. par ask mode: For non-interactive par ask, Confirm is treated as Allow (no prompting). Users can override with --confirm flag to get interactive confirmation in ask mode too.

    Estimated: 0.5 day (small delta on top of v0.4.0 §1.2 SDK hook).

    Success Criteria: - [ ] Matching bash commands show [y/n] prompt in REPL - [ ] y proceeds, n returns permission-denied error to the agent - [ ] Configurable patterns in config.json - [ ] par ask auto-allows (no TTY prompt) - [ ] 2+ new tests

    Design Decisions Log¶

    Decision Date Choice Rationale
    Persistence architecture 2026-06-09 Subscribe to event bus (not thread into engine) Engine stays pure (no I/O), natural backpressure via DLQ, same pattern as in-memory subscribers
    Session granularity 2026-06-09 One Runtime.invoke call = one session Fine-grained enough for par history, coarse enough to be useful. Matches PydanticAI/LangGraph thread_id semantics
    Handoff mechanism 2026-06-09 Typed ADT constructor (not JSON synthetic tool) STRATEGY Q4 priority #1 = type rigor. Python frameworks use duck typing; OCaml needs explicit typed signal
    Handoff scope 2026-06-09 Single-chain only (A→B), no tree branching STRATEGY §7 still excludes full session tree. Narrow exception for pre-registered agent delegation
    Structured output 2026-06-09 Middleware on register_tool (not GADT generic result) Reuses existing bash validation pattern. No type-system gymnastics. 2 days vs 2 weeks
    Event schema 2026-06-09 Keep flat events table + add session_id + actions_json columns Oracle: don't introduce LangGraph 3-table design — PAR already has workflow_states for checkpoints. Flat log + session index is sufficient for v0.4
