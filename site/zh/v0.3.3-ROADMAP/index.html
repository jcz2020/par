    P-A-R v0.3.3 开发路线图¶

    版本: v0.3.3 状态: 🆕 规划阶段 目标读者: LLM 后端工程师（部署性 > 学术性） 前置版本: v0.3.2（文档翻译 + zh 镜像，666 OCaml + 16 Python 测试通过） 核心定位: "品质加固 + Python 对齐" — 测试补洞、技术债清零、Python FFI 扩张

    🎯 v0.3.3 一句话定位¶

    v0.3.0 打了 SDK 基础，v0.3.1 加了 bash + MCP，v0.3.2 翻译了文档。 v0.3.3 补四件事：CLI 集成脱节（对话记忆/bash 空壳/无 MCP）、13 个模块零测试覆盖、8 项 OPS 技术债、Python 绑定落后 OCaml SDK 30+ 子模块。

    • ✅ 100% backward-compatible（不删 API、不改已有函数签名、不动 Runtime 签名）
    • ✅ 纯 additive（新增 ?conversation 参数、新增测试文件、扩展 C FFI 面、Python 新增方法）
    • ✅ 11-15 天可完成

    概述¶

    v0.3.3 = CLI 集成修复（CLI）+ 测试覆盖补洞（TEST）+ OPS 技术债清零（DEBT）+ Python FFI 扩张（PYFFI）。

    为什么四件事要一起做： - CLI 对话记忆是最影响用户体验的 bug → W0 优先修 - 清 OPS 债（validation guards）需要测试验证 → W1 先清债，W2 补测试覆盖 W1 的修法 - Python FFI 扩张需要 C FFI 新增函数 → 新增函数需要 OCaml 测试 → W2 先测 OCaml，W3 再扩 Python - 四件事形成依赖链：CLI → DEBT → TEST → PYFFI → RELEASE

    来源标注： - CLI：用户实际使用 par REPL 发现的 3 个集成 bug（无对话记忆、bash 空壳、无 MCP） - DEBT：v0.3-ROADMAP §"Known Tech Debt"（13 项，OPS-2/3/4/7/10 已修，剩 8 项） - TEST：代码扫描发现 13 个模块零覆盖（~2,189 行生产代码） - PYFFI：Python 绑定 vs OCaml SDK 表面对比（30+ 子模块 vs 11 个 C FFI 函数）

    ✅ W0: CLI — CLI 集成修复¶

    目标: 修复 CLI 跟 v0.3.0/v0.3.1 功能脱节的 3 个集成 bug。

    Bug 分析¶

    CLI-1: 无对话记忆（P0）¶

    现象: 用户说"你好"后再问"我刚刚说了什么"，LLM 回答"这是我们的第一条消息"。

    根因: Engine.run_agent 每次调用都通过 make_conversation（engine.ml:102）创建全新会话 [system; user]。CLI REPL 循环调用 Runtime.invoke ~message:line，但 API 只接受单条 string，无法传入历史消息。

    修复方案: 在 Runtime 层新增对话历史累积机制： - 方案 A: Runtime.invoke 新增 ?conversation 参数，CLI 维护 conversation ref - 方案 B: Runtime 内部为每个 agent_id 维护对话历史，新增 Runtime.invoke_multi_turn 自动追加 - 推荐方案 A（更显式，不隐式存储状态）

    涉及文件: - lib/core/runtime.ml — invoke 函数签名扩展 - lib/core/engine.ml — run_agent 接受 ?conversation 参数 - bin/main.ml — REPL 循环维护 conversation ref，每次 invoke 后更新

    QA: 启动 par REPL，说"你好"，再说"我刚刚说了什么" → LLM 能引用之前的话

    CLI-2: bash 工具空壳（P0）¶

    现象: 用户说"使用 bash 工具获取我在 github 上的项目列表"，LLM 回答"没有 bash/shell 执行工具"。

    根因: builtin_tools.ml:1256-1258 中 bash handler 是显式 stub，返回 Error "bash tool not installed"。CLI 调用了 Builtin_tools.builtin_tools 注册了描述符，但从未调用 Runtime.install_bash_tool（需要 ~process_mgr 和 ~clock 参数来自 Eio env，CLI 的 setup_runtime 没传）。

    修复方案: 在 setup_runtime 中调用 Runtime.install_bash_tool： 

    (* 在 setup_runtime 中，Runtime.create 成功后 *)
match Runtime.install_bash_tool
    ~process_mgr:(Eio.Stdenv.process_mgr env)
    ~clock:(Eio.Stdenv.clock env)
    rt with
| Ok _ -> ()
| Error e -> Printf.eprintf "Warning: bash tool install failed: %s\n" ...

    涉及文件: - bin/main.ml — setup_runtime 加 bash install 调用 - lib/tools/builtin_tools.ml — 确认 bash stub 被覆盖（install_bash_tool 替换 handler）

    QA: par ask "用 bash 执行 echo hello" → 返回 {"stdout": "hello\n", "exit_code": 0, ...}

    CLI-3: 无 MCP 集成（P1）¶

    现象: v0.3.1 MCP 功能对 CLI 用户完全不可见。

    根因: setup_runtime 调用 Runtime.create 时没传 ~mcp_servers、~mcp_process_mgr、~mcp_clock。CLI 不读取 MCP 配置。

    修复方案: 1. 在 ~/.par/config.json 加 mcp_servers 字段 2. Par_config 加 mcp_servers 解析 3. setup_runtime 传 ~mcp_servers / ~mcp_process_mgr / ~mcp_clock

    涉及文件: - bin/par_config.ml — 加 mcp_servers 配置字段 - bin/main.ml — setup_runtime 传 MCP 参数 - lib/core/types.ml — 确认 mcp_server_config 类型可序列化

    QA: 配置 mcp_servers: [{name: "test", command: "echo", args: ["hello"]}]，启动 par，/health 显示 MCP server 状态
