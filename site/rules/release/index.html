    Release & Distribution Rules¶

    Pre-Bump Gate (MANDATORY — Run Before ANY make release-*)¶

    Before running any version bump command, answer ALL three:

    1. Public API changed? — types, .mli signatures, CLI flags, Python binding surface?
    2. User-visible behavior changed? — new feature, bug fix, changed output?
    3. ROADMAP expects a bump? — does docs/v*-ROADMAP.md plan this version?
    Answers Action
    All NO NO BUMP. Refresh beta date only: sed -i 's/-beta\.[0-9]*/-beta.'"$(date +%Y%m%d)"'/' dune-project && make sync-version
    Any YES Proceed to make release-patch/minor/major/beta per semver skill

    Violating this gate is the #1 cause of version desync from the roadmap.

    Beta-First Strategy (Mandatory)¶

    Always ship beta first. Version format: X.Y.Z-beta.YYYYMMDD (dot separator).

    Only tag/release a stable version when the user explicitly says "publish release": - Strip -beta.YYYYMMDD suffix - Tag vX.Y.Z (no pre-release suffix) - Push tag to trigger release pipeline - Update README/CHANGES with stable version and test count

    Before "publish release", these are FORBIDDEN: - Stripping version suffix - Creating stable tags (beta tags are allowed — see Beta Publishing below) - Updating docs with stable version numbers

    Beta phase allows: merge to main, update CHANGES.md, update code.

    Beta Publishing (optional — ship beta binaries for testing)¶

    Beta tags trigger the release pipeline, but GitHub Release is automatically marked as pre-release by CI (release.yml checks for -beta. / -rc. in the tag name). Users who install from GitHub Releases will see the pre-release badge and must opt in.

    # After beta development is done and tests pass:
git tag v0.4.0-beta.20260613
git push origin v0.4.0-beta.20260613
# CI builds and publishes as GitHub pre-release automatically

    Beta releases on GitHub: - Appear with a yellow "Pre-release" badge - NOT selected by latest — par update ignores them unless --pre-release is specified - Can coexist with a stable release (stable always wins for latest) - Users must explicitly opt in via GitHub UI or par update --pre-release

    Release Flow (Two Phases)¶

    Phase A: Beta (immediately after development)¶

    1. Set version to X.Y.Z-beta.YYYYMMDD in dune-project
    2. make sync-version → sync to Python bindings
    3. make install-dev → build, install, verify
    4. dune runtest + pytest → all pass
    5. Update CHANGES.md with beta entry
    6. Merge to main, push
    7. (Optional) Tag vX.Y.Z-beta.YYYYMMDD and push → CI publishes as pre-release

    Phase B: Stable (only on explicit "publish release")¶

    1. Strip -beta.YYYYMMDD suffix → X.Y.Z
    2. make sync-version + make install-dev
    3. Update docs: README badges, test count, wheel URL
    4. Update CHANGES.md: beta → stable entry
    5. Tag: git tag -a vX.Y.Z -m "Release vX.Y.Z"
    6. Push: git push origin main --tags
    7. CI builds: release.yml on ubuntu-latest + macos-15
    8. GitHub Release: binaries + checksums auto-uploaded
    9. Manual: opam publish (first time), PyPI twine (first time)
    10. Manual: macOS-x64 via scripts/build-macos-x64.sh
    11. Verify: curl -fsSL https://raw.githubusercontent.com/jcz2020/par/main/install.sh | bash && par --version

    Binary Naming¶

    Format: par-v{version}-{platform}

    Platform Build Artifact
    linux-x64 CI (ubuntu-latest) par-v{version}-linux-x64
    macos-arm64 CI (macos-15) par-v{version}-macos-arm64
    macos-x64 Manual (scripts/build-macos-x64.sh) par-v{version}-macos-x64

    Checksums: sha512-checksums.txt alongside binaries.

    macOS-x64 Manual Build (Strict Sync)¶

    scripts/build-macos-x64.sh build steps MUST match .github/workflows/release.yml exactly. Any dependency or step change must update both files simultaneously.

    End-to-End Release Test (MANDATORY before any PyPI upload)¶

    Added 2026-06-21 per v0.4.10 postmortem (docs/release-pipeline-redesign.md). Three consecutive P0 release bugs in v0.4.8/9/10 (PAR-0qf, PAR-8cs, PAR-cog) exposed the gap: no end-to-end install test gated the PyPI upload. Every release from v0.4.11 onward MUST pass this procedure.

    Why this exists¶

    Before this procedure, the release pipeline tested only "build succeeded". A wheel could ship to PyPI with: - Missing native library (par_capi.so) — invisible to build - Native library built against too-new glibc — invisible to build, breaks install on target platforms - Wheel built with bad metadata (UNKNOWN name) — invisible to build, fails pip install

    Each of these actually happened in v0.4.8, v0.4.9, v0.4.10 respectively. The end-to-end test catches all three classes by actually installing the wheel in fresh containers and exercising the API.

    Step 1 — Local dry-run (fast iteration, ~2 minutes)¶

    Before pushing any tag, run:

    bash scripts/release-local-test.sh

    This script: 1. Builds the wheel locally (make install-dev + pip wheel) 2. For each container in matrix (debian:12, ubuntu:22.04, ubuntu:24.04): - Spins up the container via Docker - Installs Python 3 + pip + venv - Runs scripts/release-acceptance-test.py <wheel> <version> 3. Reports pass/fail per platform; exits non-zero if any fail

    Requires Docker locally. If unavailable, use Step 2 (CI workflow) directly.

    Step 2 — CI acceptance workflow (final gate, ~5 minutes after PyPI publish)¶

    .github/workflows/release-acceptance.yml triggers automatically when PyPI publish workflow completes. It runs the same acceptance test script in 3 containers via GitHub Actions matrix.

    • If any platform fails: the workflow fails. The release is considered broken. DO NOT run twine upload until all platforms are green. Investigate the failure, fix, re-tag.
    • If all 3 platforms pass: the wheel is verified. Proceed to Step 3.

    Step 3 — Manual twine upload (only after Step 2 green)¶

    # Download the proven-good wheel from GH Release
gh release download vX.Y.Z --repo jcz2020/par --pattern 'par_runtime-*.whl' --dir /tmp/

# Upload to PyPI
twine upload --non-interactive /tmp/par_runtime-X.Y.Z-py3-none-any.whl

    Step 4 — Post-upload matrix verification (15 minutes)¶

    After PyPI shows the new version, verify on real target machines (not Docker, not CI):

    Platform Method Expected
    Debian 12 (or current dev box) pip install par-runtime==X.Y.Z + python -c "import par_runtime; print(par_runtime.__version__)" works, prints version
    Debian 12 full Runtime init + register_tool + close (use TEST_CONFIG from bindings/python/tests/test_runtime.py:14-45) works
    Ubuntu 22.04 LTS (CI runner OK) same works
    Ubuntu 24.04 LTS (CI runner OK) same works
    macOS arm64 install.sh + par --version prints version

    If any platform fails post-upload, immediately yank the version on PyPI and start a hotfix release.

    Test levels (defined in scripts/release-acceptance-test.py)¶

    • Level 1 (MUST pass, blocking): import par_runtime + __version__ matches expected. Catches glibc mismatch, missing .so, broken metadata.
    • Level 2 (SHOULD pass, blocking): Runtime(config) + register_tool + close. Catches ABI mismatches, config schema drift.
    • Level 3 (informational, non-blocking): log __all__ exports + actual .so path. For debugging only.

    Acceptance matrix platforms¶

    The 3 platforms are chosen to cover the glibc baselines we care about:

    Container glibc version Represents
    debian:12 2.36 Current Debian stable, common server
    ubuntu:22.04 2.35 Most popular Linux LTS, build host baseline
    ubuntu:24.04 2.39 Latest Ubuntu LTS, forward-compat check

    Wheels built on ubuntu-22.04 (glibc 2.35) should install on all 3. If ubuntu:24.04 fails, the wheel has a backward-compat issue (rare). If debian:12 fails, the wheel has a forward-compat issue (impossible if built on 22.04). If ubuntu:22.04 fails, something is fundamentally wrong.

    RHEL 8+ / Ubuntu 18.10+ / Debian 10+ covered by manylinux_2_28 (shipped v0.4.13); ARM64 Linux + ARM64 macOS added in v0.5.0 (intel Mac + RHEL 7/CentOS 7 out of scope). Users on those platforms use source build for now.

    curl -fsSL https://raw.githubusercontent.com/jcz2020/par/main/install.sh | bash  # latest
PAR_INSTALL_VERSION=vX.Y.Z curl -fsSL ... | bash  # specific version
PAR_INSTALL_PREFIX=~/.local curl -fsSL ... | bash  # custom location

    par update¶

    par update           # check and install latest
par update --check   # check only

    Flow: CDN download checksums → extract version from filenames → download binary → SHA-512 verify → Unix.rename self-replace.

    CI/CD Workflows¶

    Workflow Trigger Purpose
    ci.yml push/PR to main Build + test + version validation + test count
    release.yml tag v*.*.* Build 2-platform binaries + checksums → GitHub Release (pre-release if -beta./-rc.)
    opam-publish.yml tag v*.*.* Tarball + .opam → GitHub Release
    pypi-publish.yml tag v*.*.* Python wheel → GitHub Release

    Release Checklist¶

    In addition to the 12-item pre-release checklist in docs/rules/docs.md:

    1. install.sh GITHUB_REPO variable is correct
    2. par update correctly parses GitHub Release tag_name
    3. sha512-checksums.txt exists in Release assets
    4. Manual test: curl ... | bash && par --version && par update --check
    5. macOS-x64: run scripts/build-macos-x64.sh, verify upload

    Local Testing¶

    No network needed (after every code change): 

    make install-dev                  # build + install + verify
par --version                     # confirm version
par update --check                # fails with 404 but confirms TLS + platform detection

    With Release artifacts (after tagging): 

    PAR_INSTALL_VERSION=vX.Y.Z curl -fsSL ... | bash
par --version
par update --check

    Accept risk: - Unix.rename atomic self-replace: reliable on Linux and macOS (same filesystem) - self_path() via /proc/self/exe: Linux only, macOS falls back to Sys.argv.(0) - par update replaces the running instance location

    Branch Model¶

    Trunk-based development with short-lived release branches.

    Branch Types¶

    Branch Purpose Lifetime
    main Trunk. Always shippable. CI gates every push. Permanent
    release/vX.Y.Z Cut from main for stabilization. Tag from here. Merge back when done. Days to weeks

    Workflow¶

    1. Development → commits to main (feature branches optional for solo work)
    2. Release prep → git checkout -b release/vX.Y.Z <known-good-sha>
    3. Beta → version X.Y.Z-beta.YYYYMMDD, merge to main, push (no tag)
    4. Production → strip beta suffix, tag vX.Y.Z on main, push tag
    5. Hotfix → fix on main first, cherry-pick to release branch, tag vX.Y.Z.1

    Cherry-pick Direction¶

    Always trunk → release. Fix on main, verify in CI, cherry-pick to release branch. Never reverse.

    When Release Branches Are Overkill¶

    For solo development with fast CI, tagging directly on main is sufficient. Cut release branches when: - Multiple developers stream commits during stabilization - Maintaining multiple released versions simultaneously
