    P-A-R v0.3.0 开发路线图¶

    版本: v0.3.0 状态: 🚧 战略讨论已完成 ✅ — 进入实施规划阶段 目标读者: LLM 后端工程师（部署性 > 学术性） 前置版本: v0.2.0（198 tests passing） 参考: pi-agent-core — 58.9k stars，验证过的 agent core 设计

    ⚠️ 关键边界声明¶

    PAR 是 SDK 库，不是部署框架。

    • ✅ SDK 提供机制（mechanism）：暴露内部状态查询接口（Runtime.health()、Runtime.metrics()）
    • ❌ SDK 不提供策略（policy）：不假设用户用什么部署环境（k8s / systemd / bare metal / 嵌入式 wrapper）
    • ❌ SDK 不内置 HTTP server
    • ❌ SDK 不绑定任何部署形态

    用户怎么暴露 health 端点、怎么部署、怎么监控——是用户的责任。PAR 只负责提供查询函数。

    OPS-1 系列（原提案"部署工程"）的修正：把"添加 health 端点"改成"添加 Runtime.health() 函数"。机制与策略分离。

    ✅ P0 战略定义（已完成）¶

    1. 价值定位¶

    PAR = OCaml 圈的 LangChain + LangGraph。 - MVP 友好：问答、RAG、翻译工具 - 严肃场景：8 状态机、Human-in-the-Loop、多 agent 协作 - 集成组件 + 提示词管理

    2. 目标用户¶

    LLM 后端工程师——agent 是产品一部分，重视： - 部署性（opam + PyPI 双发布） - 运行时稳定性、故障恢复 - 类型严谨（build-time error catching）

    3. 差异化主张¶

    vs LangChain (Python): - 类型严谨：LangChain 能崩的错，PAR 在 build 时就报 - Eio 轻量：同负载下 PAR 内存/CPU 仅为 Python 的 1/5 - 原生多 provider：OpenAI / Anthropic / Ollama / Custom 零成本切换

    vs pi-agent-core (TypeScript): - OCaml 生态唯一选择：他们不重叠，PAR 是 OCaml 圈唯一选项

    4. 优先级（资源分配）¶

    1. 类型严谨（最高）
    2. 并发性能（Eio 优势）
    3. 多 provider 适配（变现点）
    4. OCaml 生态定位（人口天花板）

    5. 发布策略¶

    opam + PyPI 双发布： - par — OCaml SDK（opam） - par_runtime — Python 包（PyPI，通过 ctypes）

    概述¶

    v0.2.0 已完成 198 个测试。v0.3.0 聚焦三件事：

    1. 补 v0.2 留下的债——3 个真实 bug（retry middleware 死代码、silent persistence error、cancel_task 跨 FFI 崩溃）
    2. 修 v0.2 暴露的 SEVERE 类型 gap——5 个用户实际遇到的 runtime 崩溃点
    3. 借鉴 pi-agent-core 提升交互价值——steering、onUpdate、parallel tool

    来源标注： - 🔴 P0 债：v0.2 实施时已知但未修的 bug - 🟡 UX-5：v0.2 后类型审计发现的 SEVERE gap（Q3 #1 主张"build-time error catching"目前部分不成立） - 🟡 UX-1/2/3：来自 pi-agent-core（58.9k stars，验证过的设计模式） - 🟡 UX-4：v0.2 review 时发现 system_prompt 是裸 string 的功能缺口 - 🟢 FFI-1/2：v0.2 stub 化遗留 - 🟡 OPS-1a/b：Q2 目标用户（LLM 后端工程师）需要内部可观测性 hook

    关键洞察：3 个 P0 债必须在 v0.3 启动¶

    OPS-2 / OPS-3 / OPS-4 不是"v0.3 加新功能"，是"v0.2 留下没修的 bug"：

    • OPS-2：retry.ml:94-96 明确标注 on_error 是死代码。当前 LLM 429 错误不会重试——所有 429 直接报错返回。
    • OPS-3：runtime.ml 有 7+ 处 Error _ -> () 静默忽略 persistence 错误。数据丢失用户无感知。
    • OPS-4：runtime.ml:107 cancel_task 抛 Invalid_argument 异常。Python 端调用会崩溃整个进程（跨 FFI 边界异常未捕获）。

    如果不做这 3 个修复，所有 v0.3 新功能（UX-1/2/3/4/FFI-1）都建立在已知 bug 之上。

    成功标准: - 5 个 stub 函数全部实现真路径 - Python 端 5 行代码跑通：register_tool → register_agent → invoke → 收到响应 → close - 5 个集成测试覆盖正常路径 + 错误路径（duplicate id、invalid JSON、invoke after close） - C 端字符串所有权清晰（malloc → par_free）

    UX-1: Steering / Follow-up 队列¶

    来源: pi-agent-core steer() / followUp()

    问题: PAR 的 Runtime.invoke(agent_id, message) 是单次调用返回。长跑 agent（30s+ LLM 推理 + 工具链）期间，用户无法： - Steer — 中途注入新指令（如 "Stop! Do this instead"） - Follow-up — 在 agent 跑完后挂后续任务（如 "Also summarize the result"）

    pi-agent-core 在 TypeScript 生态验证了这个模式有效（58.9k stars，多项目使用）。

    移植设计（适配 PAR 的 Eio 同步纤程范式）:

    (* Runtime API 新增 *)
type steering_queue
type followup_queue

val create_steering_queue : unit -> steering_queue
val create_followup_queue : unit -> followup_queue

(* 在 invoke 期间被 Runtime 内部轮询 *)
val poll_steering : steering_queue -> (message list, error_category) result
val poll_followup : followup_queue -> (message list, error_category) result

    集成点 (lib/core/engine.ml): - ReAct 循环每轮 tool 执行完后检查 steering 队列 - 队列非空 → 注入到对话 → 继续 LLM 调用 - 队列空 + tool 跑完 → 检查 followup 队列 - followup 非空 → 注入新消息 → 继续 - 都空 → 退出循环

    成功标准: - 单元测试：steer 队列非空时 ReAct 循环立即拉取并注入 - 单元测试：followup 队列在 loop 出口被检查 - 集成测试：Python 端调用 Runtime.steer() 真的中断正在跑的 invoke - 集成测试：Python 端调用 Runtime.follow_up() 在 invoke 返回后立即触发下一轮 - 现有 198 tests 不退化

    估时: 半天

    风险: Eio 同步纤程与队列的并发模型需要仔细设计。建议参考 pi-agent-core 的 PendingMessageQueue 实现但适配 OCaml。

    UX-2: onUpdate 工具流式输出¶

    来源: pi-agent-core AgentTool.execute 的 onUpdate 回调

    问题: PAR 的 13 个内置工具都是 string -> string 同步返回。对于长跑工具（fetch_url 抓 10MB 页面、read_webpage 解析大型 HTML、generate_password 长参数），用户看到的是"卡住 → 突然返回"。

    pi-agent-core 的解决方案：tool handler 接收 onUpdate 回调，可以在执行过程中回传部分结果。

    移植设计（最小侵入）:

    (* tool_descriptor 新增字段 *)
type tool_descriptor = {
  ...
  on_update : (string -> unit) option;  (* 工具执行进度回调 *)
}

(* Handler 类型保持不变，但支持可选更新 *)
type handler_fn = tool_call -> cancellation_token -> (handler_result * (string -> unit) option)

    集成点: - lib/providers/openai_provider.ml 已经有 SSE 流式响应 → 流式 LLM 输出已支持 - lib/core/engine.ml 的 execute_tool 函数包装 handler 调用 - 当 handler 暴露 on_update 回调，engine 将其注册到 event bus，事件类型新增 Tool_progress variant

    成功标准: - 单元测试：tool handler 调用 on_update("reading...") 触发 event bus 事件 - 集成测试：fetch_url 工具在抓取大页面时每秒发一个 progress 事件 - Python 端：Runtime.subscribe(event_type="tool_progress", callback=...) 能收到 - 现有 198 tests 不退化

    估时: 1 天

    不做的部分: - LLM 流式输出已经在 openai_provider.ml 有 SSE 实现，Python 端暂不暴露 - 不重写现有工具 handler（保持向后兼容）

    UX-3: Parallel tool execution¶

    来源: pi-agent-core toolExecution: "parallel" 配置

    问题: PAR 的 ReAct 循环在一次 LLM 响应中收到多个 tool_call 时，串行执行。LLM 经常一次吐 2-3 个 tool_call（典型场景：先 search 再 fetch）。串行执行浪费时间。

    pi-agent-core 的解决方案：executeToolCallsParallel —— preflight（参数校验 + before_tool hook）串行做，但 tool handler 执行并发。

    移植设计（需要重构 engine.ml）:

    (* 新增 tool 调度器 *)
type tool_batch = {
  preflight : (tool_call * (tool_call -> tool_call option)) list;  (* 顺序：参数校验 + before_tool *)
  execution : tool_call list;  (* 并发：handler 调用 *)
  finalize : (tool_call * handler_result -> handler_result option) list;  (* 顺序：after_tool hook *)
}

val execute_tool_batch : Eio.Switch.t -> tool_batch -> (handler_result list, error_category) result

    集成点 (lib/core/engine.ml): - ReAct 循环收齐 LLM 返回的所有 tool_call - 第一阶段：串行 preflight（调用每个 tool 的 before_tool hook，校验参数） - 第二阶段：用 Eio.Fiber.all 并发执行所有 tool handler - 第三阶段：串行 finalize（调用 after_tool hook，按 LLM 返回顺序发射 tool_result）

    成功标准: - 单元测试：2 个 tool_call 并发执行，总时间 ≈ max(t1, t2) 而非 t1+t2 - 单元测试：preflight 失败时整个 batch 不执行 - 单元测试：finalize 按 LLM 源顺序发射 tool_result（不是完成顺序） - 集成测试：3 个 fetch_url 并发执行总时间 < 串行 1/3 - 现有 198 tests 不退化

    估时: 2 天

    风险: - Eio 纤程的取消语义需要重新设计（一个 tool 失败时其他 tool 的命运） - 现有 middleware 的 on_before_tool / on_after_tool 钩子必须保留语义

    可配置: - 全局：Runtime.config.parallel_tool_execution : bool（默认 true） - 单 tool：tool_descriptor.execution_mode : [Sequential | Parallel ]（默认 Parallel） - 串行优先：若 batch 中任一 tool 标 Sequential，整个 batch 降级为串行（学 pi）

    UX-4: System Prompt 设计 ✅ 方案敲定¶

    选定方案：A + 行业最佳实践组合¶

    问题: PAR 的 agent_config.system_prompt 是裸 string，所有 6 个文档示例都是 "You are a helpful assistant." 静态文本。

    选定方案（3 决策已完成）:

    1. API 形态：方案 A（向后兼容）
    2. 新增 system_prompt_template : system_prompt_template option 字段
    3. 保留 system_prompt : string（v0.2 既有）作为 Static 简化形式

    4. 不破坏 v0.2 用户代码

    5. 模板语法：Mustache {{var}}

    6. 跨语言友好（Python Jinja2、Go template、JS Handlebars 同源）
    7. 学习成本低，LLM 后端工程师已熟悉

    8. 语法简单（不引入控制流，降低 XSS 风险）

    9. 注入变量：系统 + tool 描述

    10. {{current_time}} — UTC ISO 8601
    11. {{agent_id}} — 当前 agent 标识
    12. {{runtime_id}} — runtime 标识
    13. {{user_variables}} — 用户提供的 key-value（Map）

    14. {{available_tools}} — 已注册工具的 JSON 列表

    15. Provider 适配：单一字符串，PAR 透传

    16. 用户写一个 system_prompt_template
    17. PAR 渲染为字符串
    18. PAR 内部根据 provider 转换：
      • OpenAI: 塞入 messages[0].role = "system"
      • Anthropic: 塞入顶层 system 字段
      • Ollama: 塞入 messages[0].role = "system"
      • Custom: 用户自己决定
    19. 不暴露 system vs developer role 区别（学 LangChain / pi-agent-core / Vercel AI SDK）

    OPS-PROC-1: Persistent Process Manager（持久进程管理器）¶

    来源: "Agent-Bash-Runtime" 方案 §3.6

    问题: LLM 在 dev workflow 场景下需要启动 dev server / watch 模式进程。当前 PAR 工具都是 fire-and-forget（read/grep/ls 立即返回）。LLM 想"启动 React dev server 然后看日志"——当前做不到。

    典型场景: - 启动 Vite/Webpack dev server - 跑 npm run dev / cargo run / python -m flask run - 启动数据库（测试用 SQLite、PostgreSQL） - 跑 long-running 测试

    设计（基于该方案 §3.6）:

    (* lib/core/process_manager.ml *)

type process_id = string

type spawn_spec = {
  argv : string list;
  cwd : sandboxed_path;
  env : (string * string) list;
  max_lifetime : float;            (* 秒 *)
  max_output_buffer_lines : int;   (* 默认 2000 *)
}

type process_status =
  | Running
  | Exited of { exit_code : int; duration : float }
  | Killed of { reason : string }

type t
val create : max_total:int -> t
val spawn : t -> spawn_spec -> (process_id, error) result
val read_output : t -> process_id -> offset:int -> max_lines:int 
                 -> (string * int, error) result
  (** 返回 (新行内容, 新 offset) —— 避免重复传输 *)
val stop : t -> process_id -> signal:[`TERM | `KILL] -> (unit, error) result
val list : t -> (process_id * process_status) list
val cleanup : t -> (unit, error) result  (* 关闭所有进程 *)

    关键安全设计（与 TOOL-1g bash 复用）: - 共享 sandboxed_path 抽象（CWD 锁定） - 共享 command ADT（不能传入裸 shell string） - 共享 Policy Functor（不同 Agent 限制不同） - 共享超时（max_lifetime 默认 300s，硬性） - 共享进程组清理（killpg 防 fork 逃逸） - 审计: emit Process_spawned / Process_exited 事件

    核心集成: - lib/core/hook.ml（OPS-6 已有）支持 process_spawn 事件——用户可在 hook 里加"启动 dev server 前必须 confirm" - Runtime.config 加 max_concurrent_persistent_processes : int - 自动清理：Runtime.close 时自动 cleanup 所有进程

    工具集成: - TOOL-1h: start_process (argv + cwd + env) - TOOL-1i: read_process_output (pid + offset) - TOOL-1j: stop_process (pid) - TOOL-1k: list_processes (返回 pid 列表)

    估时: 3-5 天

    与 TOOL-1g bash 的关系: - bash 工具实现可以直接用 Process Manager 作为后端（瞬态命令 = spawn + wait + cleanup） - 避免重复实现进程管理代码 - Persistent Process Manager 是 bash runtime 的超集

    依赖: - TOOL-1g bash（共享 ADT、Policy、超时机制） - OPS-6 hook（用户确认）

    风险: - 跨平台不一致 - 需要 root 或 user namespace 配置 - 自己实现 wrapper 容易出 escape bug

    决策点: 启动 v0.3.0 后，由用户决定 v0.4 是不是要做这个。如果做——单独立项，确保 PAR 核心包仍然轻量。
