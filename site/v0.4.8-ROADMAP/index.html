    v0.4.8 Roadmap — Structured Output (Runtime.invoke_structured)¶

    Status: Draft v1. Created 2026-06-19. Theme: Single feature — typed structured LLM output via JSON Schema. Provider-native (OpenAI response_format, Anthropic tool_use hack) with graceful fallback for other providers. Type: Feature. Minor-equivalent bump from v0.4.7 (would be v0.5.0 under strict SemVer; staying in v0.4.x per project convention until v0.5 theme emerges). Predecessor: v0.4.7 (hotfix — ignore hallucinated tool_calls). Source issues: PAR-xd5 (P0), PAR-5cc (P1). Strategic alignment: STRATEGY.md §4 priority #1 = 类型严谨. This is the strongest differentiator vs Python frameworks (LangChain/PydanticAI run-time crashes → PAR compile-time + schema-validated LLM output).

    Rationale¶

    Today, Runtime.invoke returns free-text LLM output. Users who need structured data must:

    1. Hand-write a system prompt instructing the LLM to "respond in JSON",
    2. Strip markdown fences (json ... ) themselves,
    3. Call Yojson.Safe.from_string and hope the LLM obeyed,
    4. Manually validate the shape.

    Every step is a foot-gun. Worse, PAR doesn't use the native structured-output APIs of its providers — OpenAI's response_format: { type: "json_schema", ... } (Aug 2024) and Anthropic's tool_use synthesis are both unsupported. STRATEGY.md §4 priority #1 is type rigor; an OCaml LLM SDK without first-class structured output undercuts the core value proposition.

    v0.4.8 closes the gap with one new public API: Runtime.invoke_structured. It is provider-aware (uses native structured output where available, falls back to prompt+validate elsewhere), runs an in-band retry loop on schema violations, and reuses PAR's existing JSON Schema validator (added v0.4.0 for tool output validation).

    API Design¶

    Primary entry point¶

    (* lib/core/runtime.mli — new val after line 80 *)

val invoke_structured :
  runtime ->
  agent_id:string ->
  message:string ->
  response_schema:Yojson.Safe.t ->       (* JSON Schema describing desired output *)
  ?max_repair_attempts:int ->            (* default 3; feedback retries on violation *)
  ?cancellation_token:cancellation_token ->
  ?conversation:conversation ->           (* resume an existing conversation *)
  ?on_tool_event:(event -> unit) ->
  unit ->
  (structured_invoke_result, error_category * conversation) result

    Return type¶

    (* lib/core/types.ml — new type near invoke_result at line 140-143 *)

type structured_invoke_result = {
  value        : Yojson.Safe.t;   (* schema-validated JSON *)
  raw_response : llm_response;    (* original LLM reply (for debugging / token accounting) *)
  conversation : conversation;    (* updated, includes any repair messages *)
  attempts     : int;             (* 1 = happy path; >1 = repairs happened *)
} [@@deriving yojson]

    Mirrors invoke_result shape (types.ml:140-143) so callers can chain ?conversation across calls — same convention as Runtime.invoke.

    Why generic Yojson.Safe.t (not typed 'a)¶

    A higher-order invoke_structured_t ~decode:(Yojson.Safe.t -> ('a, _) result) was considered. Deferred to v0.5 — the generic API + caller-side [%of_yojson: T] covers v0.4.8 use cases without polymorphic-record gymnastics that complicate the FFI surface.

    Convenience pattern for callers¶

    type person = { name : string; age : int } [@@deriving yojson]

let get_person rt ~message =
  let schema = `Assoc [
    ("type", `String "object");
    ("properties", `Assoc [
      ("name", `Assoc [("type", `String "string")]);
      ("age",  `Assoc [("type", `String "integer")]);
    ]);
    ("required", `List [`String "name"; `String "age"]);
  ] in
  match Runtime.invoke_structured rt ~agent_id:"default" ~message ~response_schema:schema with
  | Error (cat, _) -> Error (Runtime.string_of_error_category cat)
  | Ok { value; _ } ->
    match Person.of_yojson value with
    | Ok p -> Ok p
    | Error e -> Error ("PPX decode failed: " ^ e)

    JSON Schema Validation¶

    Reuse: existing in-tree validator¶

    lib/core/validation.ml already has a partial JSON Schema validator added in v0.4.0:

    • validate_value : Yojson.t -> Yojson.t -> string list -> (unit, validation_error) result (validation.ml:107) — recursive walker
    • validate_tool_input : Yojson.t -> Yojson.t -> (unit, validation_error) result (validation.ml:162) — top-level entry; checks type, required, properties
    • validate_tool_input_result : Yojson.t -> Yojson.t -> (unit, error_category) result (validation.ml:199) — wraps as Invalid_input

    Already used at engine.ml:104-114 for register_tool ~output_schema validation. Clone this exact call pattern for structured output.

    v0.4.8 limitation (documented honestly per Oracle D4)¶

    The in-tree validator is more limited than "1-level nesting" suggests:

    • validate_value checks type/enum/min/max/length at ONE path
    • validate_tool_input recurses into properties.X via validate_value — BUT does NOT recurse into nested properties or items
    • Consequence: "1-level for objects, 0-level for arrays". A schema {"type": "array", "items": {"type": "string"}} is validated only as "is it an array" — element types unchecked.

    v0.4.8 scope: Top-level object properties validated (type/required/enum/bounds). Array items, nested object properties, and oneOf/anyOf are NOT validated in the fallback path. Native providers (OpenAI/Anthropic) validate deeply server-side — this limitation only affects Ollama/Custom fallback. Deferred to v0.5: either extend validate_value to recurse properly (and handle items, oneOf, $ref), or add jsonschema opam dependency.

    Documentation requirement (WU-8): SDK reference must state this honestly. Example callout: "The fallback path validates top-level object properties only. For nested objects or arrays, use OpenAI or Anthropic providers which validate server-side."

    Feedback Retry Loop¶

    The gap¶

    The existing Retry middleware (lib/middleware/retry.ml) wraps the on_error hook, but on_error : error_category -> handler_result option (types.ml:215) does not receive conversation — it cannot append a repair message to the conversation. Confirmed: zero matches for feedback|retry_with|repair in lib/.

    Implication: structured-output repair cannot be implemented as middleware. It must be a local loop in Engine.

    New Engine.run_structured helper¶

    (* lib/core/engine.ml — new function near run_agent (after line 204) *)

(* Pseudocode — actual implementation in WU-2 *)
let rec run_structured
    ~response_schema:Yojson.Safe.t
    ~max_repair_attempts:int
    ~(complete_fn : ... -> Yojson.Safe.t -> (llm_response, error_category) result) (* native or fallback *)
    ~agent ~conv ...
    attempt =
  (* 1. Make the LLM call (native structured or fallback with prompt directive *)
  let result = complete_fn agent.Model.model agent.tools conv response_schema in
  (match result with
   | Error cat -> Error (cat, conv)  (* LLM/network error → no retry, propagate *)
   | Ok llm_resp ->
     (* 2. Extract JSON from response (handles ```json fences for fallback path *)
     let text = llm_resp.text in
     (match Util.extract_json_from_text text with
      | Error msg ->
        if attempt >= max_repair_attempts
        then Error (Invalid_input ("JSON parse failed after " ^ string_of_int attempt ^ " attempts: " ^ msg, None), conv_with_assistant llm_resp)
        else
          (* 3a. Append assistant reply + user feedback, recurse *)
          let conv' = add_assistant_message conv llm_resp in
          let conv'' = add_user_feedback conv' ("Your previous response was not valid JSON: " ^ msg ^ ". Please respond with valid JSON matching the schema.") in
          run_structured ~response_schema ~max_repair_attempts ~complete_fn ~agent ~conv:conv'' (attempt + 1)
      | Ok json ->
        (* 3b. Validate JSON against schema *)
        (match Validation.validate_tool_input_result response_schema json with
         | Ok () -> Ok ({ value = json; raw_response = llm_resp; conversation = conv_with_assistant conv llm_resp; attempts = attempt })
         | Error cat ->
           if attempt >= max_repair_attempts
           then Error (cat, conv_with_assistant conv llm_resp)
           else
             let conv' = add_assistant_message conv llm_resp in
             let conv'' = add_user_feedback conv' (Printf.sprintf "Schema violation: %s. Please respond with JSON matching this schema." (Runtime.string_of_error_category cat)) in
             run_structured ~response_schema ~max_repair_attempts ~complete_fn ~agent ~conv:conv'' (attempt + 1))))

    Reuse points¶

    • Engine.add_assistant_message (engine.ml:175) — append LLM reply to conversation
    • New Engine.add_user_feedback — small helper, mirrors add_assistant_message shape
    • Validation.validate_tool_input_result (validation.ml:199) — schema check (reuses v0.4.0 code)
    • drain_into_conv pattern (engine.ml:217-227) — template for mid-loop conversation mutation

    Bypassing middleware¶

    The structured repair loop is local to Engine.run_structured. It does not flow through on_error. Rationale:

    • Middleware on_error cannot see conversation (signature gap, see above).
    • Structured-output violations are a different failure mode than HTTP/network retry.
    • Keeping the loop local means zero middleware signature changes — fully backward compatible.

    Retry policy reuse: when LLM call itself returns Error (Rate_limited | Timeout | External_failure, _), the existing Retry middleware handles it as usual (via Runtime.invoke's normal path). The structured loop only handles post-LLM parse/schema failures.

    Lenient JSON Extraction (new helper)¶

    LLMs (especially fallback-path providers) often wrap JSON in markdown fences or prose. New helper:

    (* lib/core/util.ml or lib/core/json_extract.ml — new module *)

val extract_json_from_text : string -> (Yojson.Safe.t, string) result
(** [extract_json_from_text text] extracts a JSON value from a free-form LLM response.
    - Strips ```json ... ``` and ``` ... ``` fences
    - Finds the first balanced {...} or [...] block
    - Returns [Error msg] if no valid JSON found *)

    Implementation sketch: 1. Strip leading/trailing whitespace 2. If starts with ```, strip first line + last line 3. Try Yojson.Safe.from_string directly; on success, return 4. On failure, scan for first { or [, then track depth to find matching close 5. Try Yojson.Safe.from_string on the substring

    Tests: 8+ cases covering plain JSON, fenced JSON, JSON in prose, malformed JSON, nested objects, arrays, empty input, multiple JSON blocks (return first).

    Design Decisions Log¶

    # Decision Date Rationale
    D1 Generic Yojson return, not typed 'a 2026-06-19 Avoid polymorphic-record gymnastics; PPX [%of_yojson: T] at call site covers typed use
    D2 Strategy B (additive complete_structured_fn field) 2026-06-19 Non-breaking for custom providers; matches existing event_bus_service pattern; capability self-declares
    D3 Feedback retry local to Engine, not middleware 2026-06-19 Middleware on_error cannot see conversation (signature gap at types.ml:215); middleware change would be breaking
    D4 1-level schema nesting only 2026-06-19 Existing validator (validation.ml:107) supports this; covers ~90% of LLM structured-output use cases; nested deferred to v0.5
    D5 Anthropic via tool_use synthesis, not prompt-only 2026-06-19 Tool_use forcing is more reliable than prompt directives; matches PydanticAI pattern; reuses existing parse_content_blocks
    D6 Generic fallback (prompt + lenient parse) for unknown providers 2026-06-19 Don't reject providers; graceful degrade. Mock provider implements structured for tests
    D7 New Structured_output_completed event for observability 2026-06-19 Pattern matches existing Llm_response_received/Tool_completed events; enables /stats tracking of repair attempts
    D8 max_repair_attempts default = 3 2026-06-19 Industry consensus (PydanticAI: 1, OpenAI structured docs: "retry up to 3x"); 3 balances cost vs success rate
