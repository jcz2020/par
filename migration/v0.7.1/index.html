    Migration Guide — v0.7.1: Deprecation Signaling¶

    v0.7.1 introduces a reusable deprecation framework so that breaking API changes no longer happen silently. Historically, breaking changes shipped in v0.6.4 through v0.6.9 (renamed functions, moved types, new required parameters) carried no compile-time or runtime signal: downstream code either kept compiling against a stale signature or failed with a hard "Unbound" error on upgrade, with nothing in between to warn the caller or point at the fix.

    This release closes that gap with three coordinated pieces:

    1. A new Deprecation module with an idempotent Deprecation.warn_once helper.
    2. A new Deprecated_api_called event variant on the Types.event bus.
    3. Retrofitted [@@deprecated] annotations on the APIs that were silently broken in recent releases, plus a consolidated upgrade table.

    Everything here is additive. No existing public type signature changes. Upgrading is: read the table, migrate the deprecated call sites, opt in to event-bus delivery if you want deprecation hits audited.

    The Deprecation module¶

    Deprecation lives in lib/core/deprecation.ml and is re-exported as Par.Deprecation. It gives every deprecated call site two signals:

    • A Logs.warn entry, fired once per process per fn_name (idempotent).
    • A Deprecated_api_called event on the registered emitter, fired the first time a given fn_name is used.

    It is thread-safe: the idempotency table and the emitter reference are guarded by an Eio.Mutex, matching the protected_hashtbl pattern used elsewhere in lib/core/.

    warn_once¶

    val Deprecation.warn_once :
  since:string ->      (* e.g. "v0.6.9" *)
  removed_in:string -> (* e.g. "v0.8" *)
  migration:string ->  (* one-line fix description *)
  fn_name:string ->
  unit -> unit

    Wire a deprecated call site like this:

    let old_api x =
  Deprecation.warn_once
    ~since:"v0.6.9" ~removed_in:"v0.8"
    ~migration:"use Runtime.install_bash_tool ~fs:(Eio.Stdenv.fs env)"
    ~fn_name:"Module.old_api" ();
  new_api x

    Pair the runtime warning with an OCaml [@@deprecated] attribute on the .mli so callers also get a compile-time signal:

    val old_api : int -> unit
  [@@deprecated "since v0.6.9, use new_api; removed in v0.8"]

    Wiring the event to the runtime¶

    The event emitter is optional by design. Outside a runtime (for example, a library consumer that has not constructed a Runtime.t yet), the event is dropped and only the log warning fires. To deliver Deprecated_api_called to the event bus, the SQLite audit log, and any downstream consumers, register the emitter once at startup:

    Deprecation.register_event_emitter (Runtime.publish_event rt)

    The callback is invoked outside the module's mutex, so it may call back into Deprecation without deadlocking.

    The Deprecated_api_called event¶

    A new variant on the Types.event ADT:

    | Deprecated_api_called of {
    fn_name : string;
    since : string;
    removed_in : string;
    migration : string;
  }
  [@@deriving yojson]

    It is fired by Deprecation.warn_once the first time a deprecated API is called in the process, and carries everything an operator needs to triage stale usage: the function name, the version it was deprecated in, the version that will remove it, and the one-line migration. The variant derives yojson, so it serializes through the FFI and persists into the SQLite event log like every other event.

    Deprecated APIs (consolidated table)¶

    These APIs now carry [@@deprecated] compile-time warnings and fire Deprecated_api_called at runtime. Each has a one-line migration. All are removed in v0.8.

    Deprecated API Since Removed in Migrate to
    Timeout.timeout_middleware v0.6.4 (PAR-19b) v0.8 agent_config.tool_timeout via Runtime.make_agent ~tool_timeout:<seconds>
    Bash_safe_command.sandboxed_path v0.6.5 v0.8 Workspace.sandboxed_path
    Bash_safe_command.sandboxed_path_to_string v0.6.5 v0.8 Workspace.to_string

    Timeout.timeout_middleware¶

    The middleware_hook contract fires only once before the tool call and cannot enforce a deadline around handler execution, so the legacy timeout middleware was a no-op. The authoritative fix is agent_config.tool_timeout, enforced by the engine via Cancellation.with_timeout inside Engine.execute_tool. The middleware is preserved as a no-op shim that emits a deprecation signal so existing programs still type-check.

    Before:

    let stack = [ Timeout.timeout_middleware ~default_timeout:30.0 ]

    After:

    let agent =
  Runtime.make_agent ~tool_timeout:30.0
    ~model:"openai/gpt-4o-mini" ()

    Bash_safe_command.sandboxed_path family¶

    In v0.6.5 the path-admission types migrated from Bash_safe_command to the new Workspace module, which became the sole authority for path admission. The old names are kept as deprecated aliases / wrappers so downstream code referencing them gets a compile-time warning and a runtime event on upgrade, instead of a hard "Unbound type/value". Update all references:

    Old New
    Bash_safe_command.sandboxed_path Workspace.sandboxed_path
    Bash_safe_command.sandboxed_path_to_string Workspace.to_string
    Bash_safe_command.sandboxed_path_of_string Workspace.admit (now takes a Workspace.workspace first)
    Bash_safe_command.sandboxed_path_cwd removed — no ambient authority; use Workspace.of_cwd

    Note: Workspace.admit has a different arity than the old sandboxed_path_of_string — it takes the workspace as the first argument. Obtain one via rt.workspace or Workspace.of_cwd ().

    Historical context: silent breaks in v0.6.4–v0.6.9¶

    The v0.6.5 "Workspace Abstraction" release renamed eight public type signatures in one pass; v0.6.9 added a required-feeling ?fs parameter to Runtime.install_bash_tool that fails at runtime if omitted. None of these emitted any compile-time or runtime signal — downstream consumers discovered them only when their code stopped compiling or their bash tool silently ran in the wrong directory. The v0.7.1 deprecation framework is the structural fix: future breaking changes get a [@@deprecated] attribute and a Deprecated_api_called event from the moment they are announced, with a clear removal version and a one-line migration.

    For SDK extension authors¶

    If you maintain a library built on par, adopt the same pattern for your own deprecated APIs:

    1. Add [@@deprecated "since vX.Y, use Z; removed in vW"] on your .mli.
    2. Call Deprecation.warn_once in the implementation body with matching since / removed_in / migration / fn_name.
    3. Register Deprecation.register_event_emitter (Runtime.publish_event rt) at startup so your deprecation hits land in the same audit stream as PAR's own.

    This keeps the deprecation surface uniform across the ecosystem.
