    迁移指南 — v0.7.1：弃用信号机制¶

    English · 简体中文

    v0.7.1 引入了一套可复用的弃用框架，使破坏性 API 变更不再静默发生。过去 v0.6.4 至 v0.6.9 中出现的破坏性变更（重命名函数、迁移类型、新增必填参数） 没有任何编译期或运行时信号：下游代码要么继续按过期签名编译，要么升级时 直接报“Unbound”错误，中间没有任何机制能警告调用方或指向修复方法。

    本次发布用三个相互配合的组件填补了这一空白：

    1. 新的 Deprecation 模块，提供幂等的 Deprecation.warn_once 辅助函数。
    2. Types.event 事件总线上新增 Deprecated_api_called 事件变体。
    3. 为近期被静默破坏的 API 补打 [@@deprecated] 标注，并附统一的升级表格。

    此处所有改动均为增量，不改变任何已有的公开类型签名。升级方式：阅读 表格、迁移被弃用的调用点、若希望审计弃用调用则接入事件总线投递。

    Deprecation 模块¶

    Deprecation 位于 lib/core/deprecation.ml，并以 Par.Deprecation 重新导出。它为每个被弃用的调用点提供两路信号：

    • 一条 Logs.warn 记录，对同一 fn_name 每进程只触发一次（幂等）。
    • 在已注册的 emitter 上抛出一个 Deprecated_api_called 事件，仅在某个 fn_name 首次被使用时触发。

    该模块是线程安全的：幂等表与 emitter 引用由 Eio.Mutex 保护，与 lib/core/ 中其他地方使用的 protected_hashtbl 模式一致。

    warn_once¶

    val Deprecation.warn_once :
  since:string ->      (* e.g. "v0.6.9" *)
  removed_in:string -> (* e.g. "v0.8" *)
  migration:string ->  (* one-line fix description *)
  fn_name:string ->
  unit -> unit

    被弃用的调用点按如下方式接入：

    let old_api x =
  Deprecation.warn_once
    ~since:"v0.6.9" ~removed_in:"v0.8"
    ~migration:"use Runtime.install_bash_tool ~fs:(Eio.Stdenv.fs env)"
    ~fn_name:"Module.old_api" ();
  new_api x

    同时在 .mli 上添加 OCaml [@@deprecated] 属性，使调用方额外获得编译期 信号：

    val old_api : int -> unit
  [@@deprecated "since v0.6.9, use new_api; removed in v0.8"]

    把事件接入运行时¶

    事件 emitter 按设计是可选的。在运行时之外（例如尚未构造 Runtime.t 的 库消费方），事件会被丢弃，仅日志告警仍会触发。若要把 Deprecated_api_called 投递到事件总线、 SQLite 审计日志及任何下游消费方，请在启动时注册一次 emitter：

    Deprecation.register_event_emitter (Runtime.publish_event rt)

    该回调在模块互斥锁之外被调用，因此它可以再次回调 Deprecation 而不会 死锁。

    Deprecated_api_called 事件¶

    Types.event ADT 上新增的变体：

    | Deprecated_api_called of {
    fn_name : string;
    since : string;
    removed_in : string;
    migration : string;
  }
  [@@deriving yojson]

    该事件由 Deprecation.warn_once 在被弃用 API 于本进程首次被 调用时触发，携带运维人员定位陈旧调用所需的全部信息：函数名、弃用版本、 计划移除版本，以及一行迁移说明。该变体派生了 yojson，因此可经 FFI 序列化， 并像其他事件一样持久化进 SQLite 事件日志。

    被弃用的 API（汇总表）¶

    这些 API 现在都带 [@@deprecated] 编译期告警并在运行时触发 Deprecated_api_called，各自附带一行迁移 说明，全部计划于 v0.8 移除。

    被弃用的 API 弃用于 移除于 迁移到
    Timeout.timeout_middleware v0.6.4 (PAR-19b) v0.8 agent_config.tool_timeout，经 Runtime.make_agent ~tool_timeout:<seconds>
    Bash_safe_command.sandboxed_path v0.6.5 v0.8 Workspace.sandboxed_path
    Bash_safe_command.sandboxed_path_to_string v0.6.5 v0.8 Workspace.to_string

    Timeout.timeout_middleware¶

    middleware_hook 契约仅在工具调用前触发一次，无法为实际 handler 执行强制 截止时间，因此旧的 timeout 中间件是个空操作。权威修复是 agent_config.tool_timeout，由引擎通过 Cancellation.with_timeout 在 Engine.execute_tool 内部强制执行。该中间件作为会触发弃用信号的空操作 shim 保留，使既有程序仍能通过类型检查。

    改之前：

    let stack = [ Timeout.timeout_middleware ~default_timeout:30.0 ]

    改之后：

    let agent =
  Runtime.make_agent ~tool_timeout:30.0
    ~model:"openai/gpt-4o-mini" ()

    Bash_safe_command.sandboxed_path 系列¶

    v0.6.5 中，路径准入类型从 Bash_safe_command 迁移到了新的 Workspace 模块，后者成为路径准入的唯一权威。旧名称作为弃用别名/包装保留，使引用它们 的下游代码在升级时获得编译期告警与运行时事件，而非直接报“Unbound type/value”。请更新全部引用：

    旧 新
    Bash_safe_command.sandboxed_path Workspace.sandboxed_path
    Bash_safe_command.sandboxed_path_to_string Workspace.to_string
    Bash_safe_command.sandboxed_path_of_string Workspace.admit（首个参数现为 Workspace.workspace）
    Bash_safe_command.sandboxed_path_cwd 已移除 — 无环境授权；改用 Workspace.of_cwd

    注意：Workspace.admit 的参数个数与旧的 sandboxed_path_of_string 不同 ——首个参数为 workspace。可通过 rt.workspace 或 Workspace.of_cwd () 获取。

    历史背景：v0.6.4–v0.6.9 的静默破坏性变更¶

    v0.6.5 的“Workspace 抽象”发布一次性重命名了八个公开类型签名；v0.6.9 为 Runtime.install_bash_tool 新增了一个形似必填的 ?fs 参数，缺省时会在 运行时报错。这些变更都未发出任何编译期或运行时信号——下游消费方只有在 代码停止编译，或 bash 工具静默地运行在错误目录时才察觉。v0.7.1 的弃用 框架正是结构性修复：今后破坏性变更从宣布那一刻起即带 [@@deprecated] 属性与 Deprecated_api_called 事件，并附明确的移除版本与一行迁移说明。

    致 SDK 扩展作者¶

    若你维护一个基于 par 的库，请对自己的弃用 API 采用同一模式：

    1. 在 .mli 上添加 [@@deprecated "since vX.Y, use Z; removed in vW"]。
    2. 在实现体中调用 Deprecation.warn_once，传入匹配的 since / removed_in / migration / fn_name。
    3. 在启动时注册 Deprecation.register_event_emitter (Runtime.publish_event rt)，使你的 弃用调用落入与 PAR 自身相同的审计流。

    这样可让整个生态的弃用面保持一致。
